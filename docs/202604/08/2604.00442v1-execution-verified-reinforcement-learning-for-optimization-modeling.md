## 摘要
利用大型语言模型（LLM）实现优化建模自动化，是通往可扩展决策智能的有前景路径。然而，现有方法要么依赖基于闭源 LLM 的代理式管线，推理延迟高；要么通过代价高昂的过程监督微调小型 LLM，这类模型往往对单一求解器 API 过拟合。受可验证奖励强化学习的启发，我们提出了执行验证优化建模（Execution-Verified Optimization Modeling，EVOM）框架——一种将数学规划求解器视为确定性的交互式验证器的执行验证学习方法。给定自然语言问题和目标求解器，EVOM 生成针对特定求解器的代码，在沙箱环境中执行，并将执行结果转化为标量奖励，通过 GRPO 和 DAPO 在生成—执行—反馈—更新的闭环过程中进行优化。这种仅基于结果的建模方式消除了过程级监督的需求，并通过切换验证环境而非重构求解器专属数据集，实现了跨求解器的泛化。实验在 NL4OPT、MAMO、IndustryOR 和 OptiBench 数据集上，结合 Gurobi、OR-Tools 和 COPT 求解器，结果表明 EVOM 的表现与或优于过程监督的 SFT，支持零样本求解器迁移，并可通过在目标求解器后端继续训练，实现高效且低成本的求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

## 一、研究背景与核心问题

- **研究动机**：  
  运筹优化模型（Operations Research Optimization Modeling）用于解决供应链、能源、金融等领域决策问题，其关键难点在于将自然语言描述转化为数学规划模型与可执行求解器代码。这一过程需要高度专业知识，限制了优化建模的自动化与可扩展性。

- **现有痛点**：  
  - **基于闭源大型语言模型（LLM）的多轮 prompting 方法**：准确性较高，但依赖昂贵的商业模型，推理延迟大。  
  - **基于监督微调（SFT）的开源 LLM 方法**：需要大量“过程级监督”数据（变量定义、约束构造、中间推理步骤），制备代价高，且易对特定求解器 API 过拟合，迁移性差。  

- **核心问题**：  
  如何在不依赖昂贵过程监督的情况下，使 LLM 学会从自然语言生成正确、可执行、并能泛化于不同求解器的数学优化建模代码。

## 二、方法论：执行验证优化建模（EVOM）

### 1. 核心思想

- 提出 **Execution-Verified Optimization Modeling (EVOM)**：  
  将求解器视作**确定性可验证器（verifiable verifier）**，通过执行生成代码并基于执行反馈（feasibility / optimality / correctness）计算奖励，驱动强化学习训练。

- **学习形式**：  
  一种“结果监督”（outcome-only supervision），即只依据最终执行结果作为信号，而非过程级推理轨迹。

- **训练闭环流程**：
  > 生成（Generate） → 执行（Execute） → 反馈（Feedback） → 更新（Update）

### 2. 技术细节与算法流程

- **输入输出机制**：  
  - 输入：自然语言问题 \( q \) 与目标求解器标签 \( s \)。  
  - 输出：结构化文本  
    \[
    y = \texttt{<think>}~ \text{推理逻辑} ~\texttt{</think>}~ \texttt{<code>}~ \text{求解器代码}~\texttt{</code>}
    \]

- **执行环境（Execution Harness）**：  
  在沙箱环境中执行生成的代码，提取返回结果 \( o = (c, \sigma, v, \ell) \)，其中包含执行标志、求解状态、目标值和日志。

- **奖励函数设计**：  
  \[
  R(y, o; a) = r_{\text{fmt}}(y) + r_{\text{ans}}(o; a)
  \]
  - \( r_{\text{fmt}} \)：输出格式奖励（标签齐备、结构合法）  
  - \( r_{\text{ans}} \)：执行结果奖励（若求解状态与目标值与真实答案匹配则得分 1，否则 0）

- **优化算法**：  
  使用无评论家强化学习（critic-free RL）：
  - **GRPO**：组内归一化优势，计算相对得分并更新策略；
  - **DAPO**：解耦剪切 + 动态采样，避免熵塌陷并跳过无效批。

- **跨求解器泛化机制**：
  - **零样本迁移（Zero-shot transfer）**：只需切换求解器环境，无需模型更新；
  - **环境切换微调（Low-cost adaptation）**：继续相同循环于新求解器，实现快速自适应。

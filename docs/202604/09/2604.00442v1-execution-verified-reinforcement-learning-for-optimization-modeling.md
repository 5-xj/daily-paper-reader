## 摘要
利用大型语言模型（LLM）自动化优化建模是实现可扩展决策智能的一个有前景的方向，但现有方法要么依赖于基于闭源LLM构建的代理式流水线（推理延迟高），要么通过代价高昂的过程监督对较小的LLM进行微调，而这种监督常常会过拟合于单一求解器API。受可验证奖励强化学习的启发，我们提出了执行验证优化建模（Execution-Verified Optimization Modeling, EVOM）框架，这是一种执行可验证的学习方法，将数学规划求解器视为确定性的交互式验证器。给定自然语言问题及目标求解器，EVOM生成求解器专属代码，在沙盒环境中执行，并将执行结果转化为标量奖励，通过GRPO和DAPO进行优化，形成“生成–执行–反馈–更新”的闭环过程。这种仅基于结果的形式避免了过程级监督的需求，并通过切换验证环境而非重建求解器特定数据集，实现跨求解器的泛化。我们在NL4OPT、MAMO、IndustryOR和OptiBench等数据集以及Gurobi、OR-Tools和COPT等求解器上的实验表明，EVOM的性能可与或优于过程监督下的SFT模型相匹配，支持零样本求解器迁移，并可通过在目标求解器后端继续训练实现高效、低成本的求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

## 2. 方法论：EVOM 框架设计与原理

### 核心思想
- 提出 **Execution-Verified Optimization Modeling (EVOM)** 框架，将“求解器视为执行验证器”，利用求解器的确定性执行结果（如可行/最优状态、数值解）作为强化学习奖励。
- 学习过程仅基于结果级监督（outcome supervision），无需中间推理步骤或人工标注的参考代码。

### 技术流程
EVOM 的学习闭环结构为：
> **生成（Generate） → 执行（Execute） → 反馈（Feedback） → 更新（Update）**

具体步骤：
1. **输入**：自然语言问题 \( q \) 与目标求解器 \( s \)；
2. **输出格式**：模型生成包含两部分结果：
   - `<think>`：数学推理与建模；
   - `<code>`：对应求解器的可执行 Python 代码；
3. **沙盒执行环境**：执行生成的代码，收集标准化的执行结果 \( o = (c, \sigma, v, \ell) \)，其中包括运行状态、求解器状态标识、目标值及日志；
4. **奖励定义**：
   - 格式奖励 \( r_{fmt}(y) \)：确保输出规范可解析；
   - 结果奖励 \( r_{ans}(o; a) \)：仅当执行产出与真实结果匹配（例如最优目标值或求解状态）时记为正奖励；
   - 总奖励：\( R(y, o; a) = r_{fmt}(y) + r_{ans}(o; a) \)
5. **策略更新**：
   - 使用两种强化学习优化器：
     - **GRPO（Group Relative Policy Optimization）**：基于组内奖励标准化的无critic优化；
     - **DAPO（Decoupled Clipping & Dynamic Sampling）**：采用非对称裁剪与动态采样，避免梯度坍塌。
6. **跨求解器泛化**：
   - **零样本迁移（Zero-shot Transfer）**：只需切换求解器执行环境即可测试迁移性能；
   - **低成本适配（Low-cost Adaptation）**：在新求解器执行后端继续运行学习循环，无需重新标注数据。

## 6. 主要结论与发现

- EVOM 在多个数据集上**匹敌或超过过程监督的SFT模型**，验证了结果级监督（Outcome-based Learning）的有效性；
- 可实现 **零样本求解器迁移**（如从 Gurobi 模型直接迁移至 OR-Tools），性能不降反升；
- 支持 **低成本适配**：通过切换验证环境即可快速适应新求解器；
- RL 优化器（GRPO vs DAPO）表现差异极小，说明学习稳定；
- 人工评测显示，该方法在“约束建模正确性”和“目标函数推理能力”方面显著优于过程监督基线；
- 小模型经执行验证强化学习后，表现接近或超过大模型；
- 错误分析显示，RL 最显著减少了**语法与变量定义错误**。

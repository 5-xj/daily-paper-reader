## 摘要
利用大语言模型（LLM）实现优化建模自动化是通往可扩展决策智能的一条极具前景的路径。然而，现有方法要么依赖于基于闭源 LLM 构建的智能体流水线，导致推理延迟较高；要么使用昂贵的过程监督对较小规模的 LLM 进行微调，这往往会导致对单一求解器 API 的过拟合。受具有可验证奖励的强化学习启发，我们提出了执行验证优化建模（EVOM），这是一种执行验证学习框架，将数学规划求解器视为确定性的交互式验证器。针对自然语言描述的问题和目标求解器，EVOM 生成特定于求解器的代码，在沙箱环境中执行，并将执行结果转换为标量奖励，通过 GRPO 和 DAPO 在闭环的“生成-执行-反馈-更新”过程中进行优化。这种纯结果导向的表述消除了对过程级监督的需求，并通过切换验证环境而非重建特定于求解器的数据集来实现跨求解器的泛化。在 NL4OPT、MAMO、IndustryOR 和 OptiBench 数据集上，针对 Gurobi、OR-Tools 和 COPT 的实验表明，EVOM 的表现达到或超过了经过过程监督微调（SFT）的模型，支持零样本求解器迁移，并通过在目标求解器后端持续训练实现了高效且低成本的求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

*   **核心问题**：能否在不依赖昂贵过程标注的情况下，通过求解器的自动反馈，训练出既准确又具备跨求解器迁移能力的建模模型？

### 2. 方法论：EVOM 框架
*   **核心思想**：将数学求解器（Solver）视为一个确定性的、可交互的**验证器**。利用“生成-执行-反馈-更新”的闭环，通过强化学习（RL）让模型从执行结果中学习。
*   **关键技术细节**：
    *   **输入与输出**：模型接收“问题描述 + 目标求解器名称”，输出包含 `<think>`（推理过程）和 `<code>`（特定求解器的 Python 代码）的结构化文本。
    *   **沙箱执行**：生成的代码在隔离环境中运行，调用 Gurobi、OR-Tools 或 COPT 等求解器。
    *   **奖励函数设计 ($R$)**：
        *   **格式奖励 ($r_{fmt}$)**：检查输出是否符合指定的 XML 标签格式。
        *   **结果奖励 ($r_{ans}$)**：仅基于执行结果。如果代码运行成功且求解出的目标函数值与真实答案（Ground Truth）在误差范围内一致，则给予正奖励；否则为 0。
    *   **算法流程**：采用 **GRPO**（群体相对策略优化）或 **DAPO**。GRPO 的优势在于无需额外的价值模型（Critic），通过同一问题的多个采样输出进行相对优势估计，非常适合具有确定性验证信号的场景。
    *   **环境切换**：通过更换执行后端的求解器，模型可以学习不同 API 的语法，实现低成本适配。

### 3. 实验设计
*   **数据集**：使用了四个主流基准测试：**NL4OPT**（线性规划）、**MAMO**（数学建模能力）、**IndustryOR**（工业实际场景）和 **OptiBench**（综合优化基准）。
*   **求解器环境**：涵盖了商业求解器 **Gurobi**、**COPT** 以及开源工具 **OR-Tools**。
*   **对比方法**：
    *   **闭源模型**：DeepSeek-R1, OpenAI o1, GPT-4o (含 CoT, CoE, OptiMUS 智能体系统)。

### 4. 资源与算力
*   **硬件环境**：实验在 **4 张 NVIDIA A800 GPU** 上进行。
*   **软件框架**：使用了开源强化学习框架 **verl**。
*   **模型规模**：主要实验基于 **Qwen2.5-7B**，同时也测试了 0.5B、1.5B 和 3B 版本。
*   **训练细节**：学习率 $1 \times 10^{-6}$，Batch Size 为 64，Group Size 为 8，训练 1 个 Epoch。

### 5. 实验数量与充分性
*   **实验规模**：论文进行了多维度实验，包括在 4 个数据集上的性能对比、针对 3 种求解器的适配实验、跨求解器的零样本迁移实验、以及针对不同模型规模的敏感性分析。
*   **消融实验**：专门探讨了推理块（`<think>`）对结果的影响，证明了即使是纯结果监督，中间推理过程对逻辑对齐也至关重要。
*   **客观性**：通过引入严苛的误差阈值（$10^{-4}$）和人工评估（针对约束条件和目标函数的准确性），确保了评估的深度和公平性。

### 6. 主要结论与发现
*   **性能卓越**：仅使用结果监督（Answer-only）的 EVOM 在多个基准上达到或超过了使用昂贵过程监督（Process-supervised）的 SFT 模型。
*   **强大的迁移能力**：在 Gurobi 上训练的模型在 OR-Tools 上表现出极强的零样本迁移能力，而 SFT 模型在切换求解器后性能往往会崩溃（因为 SFT 强行记住了特定 API 模板）。
*   **低成本适配**：对于新求解器（如 COPT），只需少量（约 100 条）冷启动数据配合 RL 训练，即可快速达到高性能，无需重建大规模标注数据集。
*   **推理的重要性**：移除 `<think>` 块会导致性能大幅下降，说明 RL 成功诱导模型利用中间空间进行逻辑规划。

### 7. 优点
*   **数据效率高**：摆脱了对中间推导步骤标注的依赖，只需“问题-答案”对即可训练。
*   **泛化性好**：通过求解器反馈学习的是通用的数学建模逻辑，而非死记硬背代码语法。
*   **实用性强**：支持环境切换，能够灵活应对企业从商业求解器向开源求解器迁移的需求。

### 8. 不足与局限
*   **冷启动依赖**：对于模型预训练阶段完全未见过的求解器，如果初始采样成功率为 0，RL 将无法启动，仍需少量 SFT 数据引导。
*   **复杂约束挑战**：虽然在变量类型和 API 调用上进步明显，但在处理极其复杂的隐含逻辑约束时，纯结果反馈可能不如细粒度的过程引导有效。
*   **求解器依赖**：训练过程高度依赖求解器的可用性和运行速度，对于计算极其耗时的超大规模优化问题，反馈循环可能会变慢。

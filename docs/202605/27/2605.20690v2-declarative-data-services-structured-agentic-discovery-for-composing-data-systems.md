## 摘要
智能体发现已表明，基于LLM驱动的搜索能够在基准条件下发现新颖的算法、设计和代码。将该范式迁移到多系统数据后端则面临一个更困难的问题：搜索空间是异构的，验证器是部署的堆栈是否实际运行，且组合知识在预训练中未被均匀捕获。无界智能体发现（一种根据失败日志反馈迭代的编码智能体）即使添加了迭代和显式组合知识，也无法一致地收敛到可运行的堆栈。我们提出声明式数据服务（DDS），一种从声明式用户意图中结构化智能体发现数据系统组合的架构。该框架在连续层次（意图、操作DAG、每个系统的技能、运行时归因）拥有四个类型化契约，将全局搜索分解为有界的子搜索；子智能体搜索每个类型化空间，而框架提供通道，使知识作为内联技能引用向前流动，错误作为类型化信号向后传递。作为交易后端工作负载上的可行性证明，DDS在无界发现无法收敛时成功收敛；运行时失败成为技能补丁，被下一次部署内联引用。我们将其定位为一个早期原型，报告从真实世界数据系统组合中获得的经验教训。

## Abstract
Agentic discovery has shown that LLM-driven search can find novel algorithms, designs, and code under benchmark conditions. Translating the paradigm to multi-system data backends surfaces a harder problem: the search space is heterogeneous, the verifier is whether a deployed stack actually runs, and composition knowledge is unevenly captured in pretraining. Unbounded agentic discovery, a coding agent iterating on failure-log feedback, fails to converge consistently on a working stack even when iteration and explicit composition knowledge are added. We propose Declarative Data Services (DDS), an architecture for structured agentic discovery of data-system compositions from declarative user intent. The framework owns four typed contracts at successive layers (intent, operator DAG, per-system skills, runtime attribution) that decompose the global search into bounded sub-searches; sub-agents search each typed space, while the framework provides the channels by which knowledge flows forward as inline skill citations and errors route backward as typed signals. As a proof of life on a trading-backend workload, DDS converges where unbounded discovery does not; runtime failures become skill patches that the next deployment cites inline. We position this as an early prototype reporting lessons from real-world data-system composition.

### 1. 论文的核心问题与整体含义（研究动机和背景）

随着 LLM 驱动的智能体（agent）在单仓库代码生成、算法发现等任务上取得突破，研究者开始将其应用于更复杂的**多系统数据后端组合**场景——例如一个包含消息队列、OLAP/OLTP 存储、缓存、搜索引擎等异构系统的实时交易后端。然而，将这种“无界智能体发现”范式直接迁移到该系统组合问题时，暴露了三个根本困难：

- **搜索空间异构**：不同系统（Kafka、ClickHouse、PostgreSQL、Redis 等）的配置、连接方式、故障模式差异巨大；
- **验证器是“部署运行”而不是基准答案**：智能体要生成的不是一段代码，而是一套能真正启动并通过端到端功能测试的部署栈；
- **组合知识稀缺且易过时**：哪些系统能协同工作、哪些配置是反模式，这些知识在 LLM 预训练数据中覆盖不均，且随版本迭代快速变化。

实验表明，即使引入迭代反馈循环（最多 5 轮）和显式组合知识，无界智能体方法依然无法一致收敛到可运行的多系统堆栈。论文由此提出 **DDS 架构**，通过结构化契约将全局搜索分解为有界子搜索，使智能体能在可验证、可归因、可编辑的框架内完成任务。

### 2. 论文提出的方法论：核心思想、关键技术细节

**核心思想**：框架（framework）和子智能体（sub-agent）分离，框架拥有每一层的类型化契约及验证规则，子智能体负责在各契约划定的有界子空间内进行知识密集型搜索。

**四层类型契约**：

- **L1 – 意图契约**：将用户自然语言需求转化为一个包含 6 个维度（数据模型、访问模式、规模、延迟、一致性、成本）的声明式意图；框架验证其格式并检查冲突。
- **L2 – 算子 DAG**：将意图映射为一个类型化的算子有向无环图（如 INGEST → QUEUE → STORE → SERVE），框架检查每条路径的可达性和端到端 SLO 的可满足性，在代码生成前就拒绝不可行的拓扑。
- **L3 – 技能契约**：每个数据系统对应一个结构化的 YAML 技能文件，包含 capabilities、compositions、anti_patterns、operational 四个板块；框架在规划时利用反模式等知识过滤候选产品，且技能是持久化、可编辑的，运行时失败可作为补丁写入技能。
- **L4 – 运行时归因循环**：部署后收集运行时信号（如镜像拉取失败、端口冲突、DDL 类型不匹配等），经规则分类器**类型化**并**路由到拥有违反决策的层**（L1 意图、L2 拓扑、L3 技能、L4 代码）。信号类型决定补丁方式：框架自动修复代码/主机环境错误，L3 技能补丁需人工审核，L1/L2 修改需用户确认。

**技术流程**：用户以自然语言陈述需求 → 子智能体经过 L0 询问生成草案 → L1 意图验证 → L2 算子 DAG 规划与验证 → L3 基于技能契约选择具体产品与配置 → L4 生成部署工件（Docker Compose、DDL、数据生产者等）并启动 → 运行时信号归因反馈，形成闭环。

### 3. 实验设计：数据集 / 场景、Benchmark、对比方法

**场景**：主要案例是一个**交易后端**（实时市场数据接入、时间序列分析、低延迟仓位查询、预算 300 美元/月）。另有描述性第二场景（**聊天平台**）侧重算子代数扩展性。

**实验变量**（n=10 每组）：
- **A**：仅自然语言需求，无框架，无指定系统（无限发现基线 + 5 轮外循环）。
- **B**：需求 + 显式列出系统名（Kafka、ClickHouse、PostgreSQL、Redis），其余同 A。
- **C**：需求 + 将 DDS 技能 YAML 作为纯文本粘贴到提示中（含组合知识但无结构化通道）。
- **DDS one-shot**：使用完整 DDS 框架，但无外循环（仅内部 T0 验证）。
- **DDS + 5-iter**：DDS 框架 + 相同外循环（5 轮迭代）。
- **技能消融**（n=3/变体）：全技能 / 删除 operational 和 anti_patterns / 仅保留 capabilities。
- **学习循环**：第一次部署后遇到 4 个真实失败 → 按归因打补丁（仅技能 + 主机策略编辑） → 第二次部署验证修复效果。
- **归因分类测试**：对 5 个已知故障类（镜像拉取失败、端口冲突等），每类注入 4 次（n=20），验证规则分类器是否准确归因。

**评估层级**：
- T0：所有生成工件语法正确。
- T1：docker compose up 后所有服务健康检查通过。
- T2：端到端烟雾查询返回预期行。

**对比对象**：A、B、C、DDS（+/- 外循环）。

**结果**见 Table 3：
- DDS + 5-iter 在所有测试中 10/10 通过 T0/T1/T2，中位数成本 $1.94，中位数 44 轮次。
- 条件 A 仅 2/10 通过 T1，成本 $5.76，268 轮次。
- 条件 C（加技能文本）6/10 T1，4/10 T2，成本 $5.06。
- 技能消融：去除 operational + anti_patterns 后 T1 降为 0/3。

### 6. 论文的主要结论与发现

1. **DDS 一致收敛，无界发现不行**：在外循环框架下，DDS 在 10/10 次达到 T1/T2，而原始无界代理（A）仅 2/10 达到 T1。DDS 的中位数成本（$1.94）仅为 A（$5.76）的 1/3，轮次（44 vs 268）约为 1/6。
2. **结构（类型契约）是获胜的关键，而非迭代本身**：即使将技能知识填入提示（条件 C），T1 率仅提升至 6/10，成本仍高于 DDS。去除了结构化契约后，同样的知识不起作用。
3. **技能契约中操作性和反模式内容起主要作用**：剥离 operational 和 anti_patterns 后，T1 从 2/3 降至 0/3；仅保留 capabilities 无实际帮助。
4. **运行时失败可被归因并转化为持久技能补丁**：第一次部署后的 4 个实际错误（端口冲突、镜像缺失、DDL 类型不匹配、库缺失）均被正确归因到 L3 或主机策略层；通过编辑技能 YAML（无代码改动）在第二次部署中全部修复。
5. **无界发现失败的两个结构模式**：①下游失败触发架构级重写，导致之前通过的部分回归；②失败日志没有层归属，智能体猜测错误子搜索，进入无进展循环。

### 7. 优点：方法或实验设计上的亮点

- **创新架构**：四层类型契约设计将复杂的问题分解为可验证、可归因、可编辑的子问题，为多系统组合的自动化提供了系统化的解决路径。
- **严格的实验对比**：在同一模型、工具、迭代上限下，系统比较了“无框架-有框架”“无迭代-有迭代”“有纯文本知识-有结构化知识”多种条件，实验设计干净、结论可信。
- **真实部署验证**：验证器不是基准答案，而是实际 docker compose 启动和端到端查询通过，贴近工业实践；还展示了实时数据流（Coinbase 喂数据）的可行性。
- **学习循环演示**：展示了如何从实际失败中积累知识（技能补丁），使框架具备持续改进能力。
- **开源 Benchmark 基础**：设计了 DDS-Bench（6 个意图、5 个领域、30 个工作负载、59 个探针模板），为后续研究提供标准化评价平台。

### 8. 不足与局限

- **实验覆盖有限**：仅在一个领域（交易）上完成量化验证，第二领域（聊天）为描述性，未提供对比数据。模型仅用了 Claude Opus 4.6，未测试其他 LLM。
- **样本量小**：主对比每组 n=10，消融组 n=3，统计强度不足，部分结果（如 DDS one-shot 与 DDS+5-iter 的 T1 率差异）可能受随机波动影响。
- **归因分类器的泛化性未知**：仅测试了 5 个手工规则覆盖的故障类（n=20，均为分布内），对未见过的错误类型（如 TLS 握手失败）如何处理未评估。

- **技能需手工编写**：当前技能库由人撰写，未实现自动抽取，其维护成本在系统快速迭代时可能较高。
- **安全与治理**：论文注意到安全（如加密）是横切关注点，但未在架构中系统处理，在聊天案例中仅是观察到其重要性。

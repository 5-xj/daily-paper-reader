## 摘要
LLM引导的进化搜索（进化系统）在数学和组合任务上取得了最先进的结果，但大多数现有系统仅报告多次运行中的最佳结果，而未记录运行之间的分布。我们提出一个问题：固定预算的LLM调用应如何分配，以及单次运行达到报告数字的可靠性如何。通过在五个模型和三个任务上扫描深度-广度网格，我们识别出两个经验规律：一个是适应度-计算包络线，沿该线能力排序在很大程度上取决于有效FLOPs；另一个是双线性深度-广度拟合，具有任务特定的交互作用；两者都受模型-任务能力的门控。受这些规律的启发，我们提出了BaSE（基于老虎机的自我进化），这是一种多臂老虎机，用于在并行轨迹之间分配LLM调用。在不改变模型、提示或评估器的情况下，BaSE在8个（模型，任务）单元上将平均适应度提高了12.3%，超过了最强的岛屿协议基线，在高方差设置中获得最大收益：仅通过分配即可获得可靠性提升。

## Abstract
LLM-guided evolutionary search (Evolve systems) has reached state-of-the-art results on mathematical and combinatorial tasks, yet most existing systems report only the best of many runs and leave the run-to-run distribution undocumented. We ask how a fixed budget of LLM calls should be allocated, and how reliably a single run reaches the reported numbers. Sweeping the depth-breadth grid over five models and three tasks, we identify two empirical regularities: a fitness-compute envelope along which capability ordering largely collapses on effective FLOPs, and a bilinear depth-breadth fit with task-specific interaction; both are gated by model-task capability. Motivated by these regularities, we propose BaSE (Bandit-based Self-Evolving), a multi-armed bandit that allocates LLM calls across parallel trajectories. Without changing the model, prompt, or evaluator, BaSE improves mean fitness by 12.3% over the strongest island-protocol baseline across 8 (model, task) cells, with the largest gains on high-variance settings: a reliability gain from allocation alone.

## 1. 论文的核心问题与整体含义（研究动机和背景）

LLM 引导的进化搜索（Evolve 系统）在数学发现和组合优化任务上达到了最先进的结果（如 FunSearch、AlphaEvolve）。然而，现有系统普遍存在两个问题：

- **报告偏差**：大多数系统只报告多次运行中的最佳结果，而不记录运行间的分布；实际部署中用户更关心单次运行的期望性能。
- **计算成本差异巨大**：不同报告的单次运行 LLM 调用数从约 150 到 204,800 不等（跨越两个数量级），导致头部数字难以在不同系统间公平比较。

因此，论文提出核心问题：**在固定预算（固定数量 LLM 调用）下，应如何分配计算资源以平衡探索和利用（即深度 vs. 广度）？** 以及 **这种分配能否在更大程度上提高单次运行的可靠性与期望性能？** 这不仅是一个模型选择或提示设计问题，而是一个独立的计算分配优化问题。

## 2. 论文提出的方法论

### 核心思想

将进化搜索视为一个迭代优化过程，在固定预算 C 次 LLM 调用下，通过选择每次调用的父代采样策略来控制深度（T 个世代）与广度（每代 N 个并行子代）的比例（C = T × N）。论文首先描述了两种极端的父代采样协议：
- **Greedy 协议**：始终选择当前最佳解作为父代，每代生成 N 个子代（深度大，广度小）。
- **Island 协议**（如 FunSearch 所用）：维护一个划分成多个“岛屿”的种群数据库，每代按规则选择岛屿并均匀采样父代（广度大，深度浅）。

进一步，论文提出在线自适应分配算法 **BaSE（Bandit-based Self-Evolving）**。

### 关键技术细节

1. **经验规律发现（先验离线分析）**：
   - **适应度-计算包络线**：以有效 FLOPs（而非简单调用次数）为成本轴，绘制每个模型可达到的最佳适应度 Vmax(C)。发现不同模型在该包络线上大致重叠，说明模型能力差异（在足够强的模型间）主要由 FLOPs 解释。
   - **双线性深度-广度拟合**：对次饱和区域（V<0.97），用双线性回归拟合 log(1-V) = β0 + a log T + b log N + c log T log N。系数 c 决定了最优分配的位置：|c| 小（≈0）时，最优深度在角落（全深度）；|c| 大负时，最优深度在内陆（需要平衡深度-广度）。实验发现 Circle Packing 偏向全深度（c≈0），而 MinMaxDist 偏向平衡分配（c 显著为负）。

2. **BaSE 算法（在线自适应分配）**：
   - 将 K 个并行轨迹视为多臂老虎机的“臂”。
   - 每步决策：由 MAB 策略（如 UCB、EXP3.P、Thompson Sampling）选择下一个要消耗一次 LLM 调用的轨迹。
   - 被选中的轨迹内仍使用父代采样（如 greedy）进行一步进化。
   - 整体预算 C 固定，BaSE 仅改变计算在轨迹间的分配，不改变轨迹内部的进化规则。

### 公式/算法流程（文字说明）

- 固定预算 C，初始化 K 个轨迹，每个轨迹分配一次调用。
- 后续 C-K 次调用：每步用 MAB 选择一条轨迹 i，执行一次进化（从轨迹 i 的历史中选择父代，LLM 生成子代，评估），更新轨迹 i 的状态和 MAB 的奖励统计。
- 最终返回 K 个轨迹中发现的最高适应度解。

## 3. 实验设计

### 任务（数据集/场景）

三个几何优化任务，来自 AlphaEvolve/OpenEvolve 示例套件：
- **Circle Packing (CP, n=26)**：在单位正方形内放置 26 个不相交圆，最大化半径和。归一化：sum_radii / 2.635。
- **MinMaxDist (MMD, n=16)**：在平面放置 16 个点，最大化最小距离与最大距离比的平方。归一化：((dmin/dmax)^2) / (1/12.889266112)。
- **Heilbronn Triangle (HT, n=11)**：在等边三角形内放置 11 个点，最大化由任意三点构成的最小三角形面积。归一化：min_area_normalized / 0.036529889880030156。

## 6. 论文的主要结论与发现

1. **存在适应度-计算包络线**：以有效 FLOPs 为成本单位时，不同模型（如 Qwen3 8B/14B）在相同 FLOPs 下达到相近的适应度，即能力顺序几乎被 FLOPs 吸收。这支持了计算分配而非模型大小是主要决定因素的论断。

2. **深度-广度分配呈双线性规律**：在次饱和区域，适应度与深度、广度及其交叉项对数线性相关，交叉项系数 c 决定了最优分配形状（角落 vs. 内陆）。任务特质（如算法族稀疏性）是主要驱动因素。

3. **BaSE 显著提升期望适应度**：在 8 个（模型，任务）组合上，BaSE（如 Thompson）平均提升最佳适应度 12.3%（相对于最强的 Island 基线 ShinkaEvolve），**且无需改变模型、提示或评估器**——这是纯粹的分配改进。

4. **BaSE 在高方差设置中尤为有效**：对于 Heilbronn Triangle 等存在严重分布外结果的困难任务，BaSE 将系统从经常停滞在 0.0 的状况中解救出来，实现了数个百分点的提升（如 Qwen3-8B HT 从 0.7379 → 0.8736）。

5. **阈值到达效率更优**：BaSE 达到固定适应度阈值所需的世代数或 FLOPs 通常更少，尤其在 MMD 和 HT 上表现显著。

## 8. 不足与局限

- **任务多样性有限**：仅测试了 3 个几何优化任务，未涉及组合优化（如旅行商）、程序合成或问答推理。结论能否推广到更广泛的进化搜索场景未知。
- **模型覆盖偏窄**：主要使用 Qwen3 系列和 Llama-3.1-8B；未包含更大的模型（如 Qwen3 32B/72B）或闭源模型（如 GPT-4），因此 FLOPs 包络线的“能力排序崩溃”可能只在相近能力区间成立。
- **提示工程的影响未被完全隔离**：虽然论文努力保持 BaSE 与 OpenEvolve 的提示一致，但 ShinkaEvolve 的提示在 CP 和 HT 上包含了关键策略提示（如 scipy 或 degeneracy warning），这可能是其强基线的原因。论文承认这一点，但未量化提示质量对基线性能的贡献。
- **BaSE 在饱和任务上增益有限**：当某任务已被接近天花板（如 CP）时，任何分配方法提升空间很小，BaSE 与基线几乎持平。
- **臂池大小选择敏感**：臂池大小 K 需要手工设定，过大（如 K=50）可能导致每一轨迹太浅，过小则多样性不足。论文给出了经验指导（K∈{5,20}），但未提出自适应调整策略。
- **未讨论 LLM 调用失败/重复的成本**：部分 LLM 调用返回无效结果（如 HT 中 DE 出的点超出三角形），这些调用仍消耗预算。BaSE 并未显式处理此类失败，可能在低能力模型上浪费资源。
- **计算资源评估不够彻底**：未报告总实验 GPU 小时数，也未讨论 BaSE 在线分配本身的额外开销（MAB 计算量很小，但未估算）。

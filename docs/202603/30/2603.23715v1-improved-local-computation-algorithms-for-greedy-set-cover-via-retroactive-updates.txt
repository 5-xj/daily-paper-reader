Abstract 

In this work, we focus on designing an efficient Local Computation Algorithm (LCA) for the set cover problem, which is a core optimization task. The state-of-the-art LCA for computing 

O(log ∆) -approximate set cover, developed by Grunau, Mitrović, Rubinfeld, and Vakilian [SODA ’20], achieves query complexity of ∆O(log ∆) ·f O(log ∆ ·(log log ∆+log log f )) , where ∆ is the maximum set size, and f is the maximum frequency of any element in sets. We present a new LCA that solves this problem using f O(log ∆) queries. Specifically, for instances where f = poly log ∆ , our algorithm improves the query complexity from ∆O(log ∆) to ∆O(log log ∆) .Our central technical contribution in designing LCAs is to aggressively sparsify the input instance but to allow for retroactive updates . Namely, our main LCA sometimes “corrects” deci-sions it made in the previous recursive LCA calls. It enables us to achieve stronger concentration guarantees, which in turn allows for more efficient and “sparser” LCA execution. We believe that this technique will be of independent interest.

With the prevalence of large volumes of data, our perception of what makes an algorithm efficient has evolved significantly. In scenarios where accessing the entire input is impractical or even infeasible, the focus has shifted toward designing algorithms that use sublinear resources – particularly time and space – to solve a given problem. Alon, Rubinfeld, Tamir, Vardi, and Xie [RTVX11, ARVX12] introduced the Local Computation Algorithm (LCA) model to formalize the study of algorithms that, rather than processing the entire input, access only a “small” fraction of the input to produce a “small” fraction of the solution. The complexity of an LCA is then measured by the number of queries it makes to the input. For example, instead of computing the entire maximal independent set (MIS) of a graph or finding a complete set cover, an LCA provides localized solutions by answering questions such as “Is a given vertex in the MIS?” or “Is a specific set part of the set cover?” Ideally, LCAs are significantly faster than algorithms that compute the entire MIS or set cover, particularly for large inputs. Since their inception, the development of LCAs has been an active research area. In par-ticular, extensive research has focused on LCAs for a broad range of graph problems, including local reconstruction of graph properties [KPS13], maximal independent sets [RTVX11, ARVX12, BEPS16, LRY17, EMR14, Gha16, GU19, Gha22], coloring [CFG +19, DK23], algorithmic Lovász local lemma [AGI20, BGR21], approximate maximum matchings [NO08, YYI09, MRVX11, MV13, LRY17, GU19, KMNT20, BRR23], k-spanners [BCPR24, LL18, PRVY19, ACLP23], graph orienta-tion [MRS24], correlation clustering [BCMT22, DMM24], local reconstruction of graph properties [CGR13], and local generation of large random objects [MRVX11, ELMR17]. LCAs are of interest in broader algorithmic contexts: Several works demonstrate novel applications of LCAs in developing faster dynamic matching algorithms [BKS23, Beh23, BKSW24] and in the study of proper learning of monotone functions [LRV22, LV23]. Important connections have also been established between LCAs and Differential Privacy [JR13, LLRV23]. 

Local simulation of distributed algorithms. Developing LCAs by leveraging distributed algo-rithms as their starting point has been a particularly fruitful line of research. Parnas and Ron [PR07] observed that, on graphs with maximum degree d, the output of a vertex in an r-round LOCAL 

algorithm depends on at most dr vertices within its r-hop neighborhood. This idea directly trans-lates a LOCAL to an LCA with a query complexity of O(dr). Over the last few years, we have seen a proliferation of ideas on sparsifying this r-hop neighborhood to achieve more efficient LCAs. Notably, a series of works on designing LCAs for MIS [NO08, YYI09, RTVX11, ARVX12, RV16, LRY15, Gha16, LM +17, EMR18, GU19] culminated in a recent work by Ghaffari [Gha22], which demonstrates an MIS LCA with 2O(r) · poly log n query complexity. This improvement from dO(r)

to 2O(r) allows for the first polynomial time algorithm in d. Approximate maximum matching and vertex cover have also been extensively studied in this regime, e.g., see [KMNT20, Beh22] and ref-erences therein. 1 The LOCAL algorithms simulated in those works use r = O(log d) rounds, and hence those results yield poly (d, log n) LCA query complexity. Our work focuses on the set cover problem, a foundational topic in algorithm design and com-putational complexity. A set cover instance consists of a universe of elements E and a family of sets S ⊆ 2E . The set cover problem aims to find the smallest-cardinality collection S′ ⊆ S whose union equals E. We denote by ∆ the maximum size of a set, and by f we denote the maximum number of sets that contain any given element e. This problem is well-known to be NP-hard. In

Does there exist an LCA for O(log ∆) -approximate set cover that uses 2O(rSC ) queries? 

We answer this question affirmatively. Namely, our LCA has 2O(log ∆ ·log f ) = f O(log ∆) query com-plexity deterministically, and the approximation is provided in expectation. The theme of LCA sparsification has seen a proliferation in recent years and has been used to significantly improve the efficiency of LCAs. Our work contributes to this active research area by introducing novel sparsification techniques for set cover LCAs. While some previous set-cover LCAs also apply sparsification, our methods diverge significantly. 

Open problems. Our work makes an important step in LCA sparsification for set cover by showing how to simulate an rSC -round LOCAL algorithm with 2O(rSC ) queries. Observe that an LCA in which each vertex explores only two of its neighbors, during the simulation of an rSC -round algorithm, already visits 2rSC vertices. In that sense, our LCA quite sparsely explores the 

rSC -hop neighborhood of a vertex. It remains open whether an LCA for set cover with poly (∆ , f )

query complexity exists. It would be even intriguing to obtain any asymptotic improvement in the exponent of this complexity, e.g., an LCA using 2o(rSC ) queries.

1.1.2 Starting point: Fractional Set Cover algorithm 

Our starting point is Algorithm 1, a fractional LOCAL algorithm for the set cover problem that can be attributed to several prior works [BRS94, KMW06, GMRV20]. 3Details of Algorithm 1. The algorithm consists of log ∆ phases . In each phase, a subset of elements is covered by adding weights to the sets such that the following holds: 

• After removing the elements covered in a phase, the maximum size of the sets halves. 

• During each phase, the total weight added to the sets is at most O(OPT ), where OPT denotes the minimum (integral) set cover size. For convenience, we use the term “effective degree” of a set S, denoted by d(S), that counts the number of elements in S that are uncovered in the current phase. Observe that the “effective degree” reduces throughout the course of the algorithm, reaching zero for every set by the time the computation is over. We now describe the workings of a phase in which the maximum degree is at most ∆i = ∆ /2i−1.We say that a set S is “dense” if d(S) ≥ ∆i/2. Note that a dense set covers as many elements as any other set (upto a constant factor). Each phase contains log f iterations . In each iteration, a subset of elements is covered, ensuring that (i) weight is added only to dense sets and (ii) wt (e) ≤ O(1) 

for each element during the first iteration in which it becomes covered. These conditions ensure that the chosen sets cover nearly as many elements as any other family of sets, with only a few sets covering each element. Thus, this covering solution costs O(OPT ). A formal proof is described later (see Lemma 3.1). 

1.1.3 Introducing slack in thresholds and pruning (Section 3) Observation 1. Our first observation is that Algorithm 1 allows some slack in the thresholds at the expense of only a constant factor increase in the approximation ratio guarantee. In particular, Algorithm 1 uses two thresholds: (i) in the test of dense sets, i.e., whether d(S) ≥ ∆/2i, and (ii) in the test of whether an element is covered, i.e., whether wt (e) ≥ 1. Both conditions can be relaxed. The relaxation differs in our two algorithms. We describe the first and simpler one below. Instead of testing whether d(S) ≥ ∆/2i, an algorithm can test whether a set of t sampled elements of S contains at least 12 · t/ 2i uncovered elements; the factor of 1/2 and the variable t are chosen to increase the probability of “a correct test”. A similar relaxation can be performed for the other test. Algorithm 2 implements these relaxations. 

Why is Observation 1 not sufficient? To test whether a set S has effective degree d(S) ≥ ∆/2i

at the beginning of some iteration (i, j ), we must test whether at least 2i elements containing S are covered at the end of the previous iteration, (iprev , j prev ). Note that (iprev , j prev ) = ( i−1, log f ) if j =1 and (i, j −1) otherwise. Following the natural LCA design, the query complexity T (i, j ) to simulate the LOCAL algorithm up to iteration (i, j ) satisfies the recurrence T (i, j ) = 2 i · T (iprev , j prev ). It is important to note that this discussion does not yet consider the concentration required for accurate estimates; thus, even the above recurrence represents a hypothetical scenario. Even with this approach, the best solution achievable is T (i, j ) = 2 O(i2 log f +j), i.e., no better than Parnas-Ron reduction of ∆O(log ∆ log f ). Our second observation is essential for eliminating this term and thereby reducing the query complexity. 

Observation 2. The second observation is that if each phase of Algorithm 1 can be simulated with sufficiently high probability, then many recursive calls in the natural LCA based on Algorithm 1 can be pruned. The natural LCA design proceeds as follows: to determine the state of a set or element

1.1.5 Weaker invariants – addressing Challenge 1

bounds for the query complexity of the LCA , so the natural LCA does not seem suitable.

1.1.6 Retroactive updates – addressing Challenge 2 

To overcome this second challenge, we introduce the idea of retroactive updates. Let us first inspect the recurrence we aim for and deduce what we can afford to do given this recurrence. 1.1.7 Modifying fractional to integral (Section 6) 

We modify the analysis of [GMRV20] to show that a simple modification to the fractional algorithm outputs an integral solution. Importantly, in expectation, this modification yields an algorithm that incurs only an additional constant factor loss in the approximation. Note that we do not round the computed fractional cover; rather, a natural modification of the fractional cover solution also yields an integral cover. 

1.2 Comparison to the Most Relevant Prior Work 

The most relevant to our approach are [KMNT20, GMRV20, Gha22]. We now discuss the relation-ship between these results and ours. 

Remark: This subsection is tuned toward readers who are at least vaguely familiar with the techniques and ideas used in the prior work mentioned. 

Comparison of techniques. Our main recurrence Eq. (1) is inspired from [Gha22]. The main technical portion of [Gha22] is a method to transform the O(log ∆) LOCAL algorithm of [Gha16] so that it is amenable to efficient LCA simulations whose oracles satisfy Eq. (1). This transformation is similar to our solution to Challenge 2. The difference is that in [Gha22], the underlying LOCAL 

algorithm operates in O(log ∆) rounds with the guarantee that a constant fraction of those rounds are “good”. The guarantees of the “good” rounds are identical to each other, so one can afford to 

delay the occurrence of a good round. In our case, each phase has a slightly different measure of progress. In phase i we require that the effective sizes of sets “roughly” drop to ∆/2i. So for instance, we cannot afford to run the phases in parallel as they must occur sequentially one after the other. This is why we require the retroactive updates. Our design of the LCA and analysis of its round complexity borrows some ideas from [KMNT20]. The key idea here is to tie the query complexity to the algorithm’s output, making the analysis of the early-stopping rule easier. The argument of the approximation ratio, however, is very different. [KMNT20] study approximations of maximum matching and their approximation ratio arguments are done by relating to vertex cover. In addition, their approach is tailored to the maximum matching problem and requires novel concentration bounds. Our concentration bounds are simple 7compositions of Chernoff and the union bounds. Consequently, we hope our analysis is easier to comprehend and apply to other settings. Our results are to be directly compared to [GMRV20]. We solve the same problem with the same approximation guarantees but with better query complexity. The design of our and the LCA in [GMRV20] are considerably different. The main optimizations of [GMRV20] are in the simulations of an individual phase, with no direct pruning done across different phases. This is the reason behind the ∆O(log ∆) factor in the query complexity of [GMRV20]. A potential lesson here is that adaptively making the recursive calls seems more beneficial than the “repeated sparsification” technique in the 

LCA setting. 

1.3 Related Work 

As discussed in the introduction, LCAs have been studied in many contexts and for many problems. This section lists related work in addition to that mentioned in the introduction. Results that predate but can be viewed within the LCA model include locally (list)-decodable codes [Fei90, Lip89, GLR +91, FF93, KT00, Yek10, GRS00, AS03, STV01, GKZ08, IW97, KS, BAETS]), local decompression [MSZ05, SG06, FV07, GN06], local reconstruction and filters for monotone and Lipshitz functions [ACCL08, SS10, BGJ +12, JR13, AJMR12], satisfying assignments for k-CNF [RTVX11, ARVX12], local computation mechanism design [HMV16], and local decom-pression [DLRR13], 

Pruning. Different variants of pruning are implicit in prior works on LCAs such as [NO08, YYI09, Beh21, DMM24]. The underlying idea is that before trying to obtain the state of some node at the end of some iteration t in the algorithm, make sure to obtain its state in iterations 1, 2, . . . t − 1,in this order. We may find information about the state in some intermediate iteration t′ ≪ t that allows us to ignore the state of v after t′. Thus, a large chunk of recursive calls can be pruned. [NO08] suggested this optimization for designing LCAs from the Randomized Greedy MIS algorithm. [YYI09] analyzed the expected query cost of this optimization, and [Beh21] provided analysis for line graphs (i.e., the maximal matching problem). The analysis is quite involved and different in the two papers. [DMM24] provide an optimal LCA for the correlation clustering problem. The pruning in [DMM24] is much simpler: once the recursion tree contains too many calls, stop and make do with the information that you have. 

Vertex cover, i.e., the case of f = 2 . The set cover problem with f = 2 is equivalent to vertex cover for graphs, which has been extensively studied. The poly (∆ , log n) round MIS algorithm of Ghaffari [Gha22] already implies a poly (∆ , log n) algorithm for 2-approximate vertex cover through the classical reduction to MIS; that is, maximal matching is a 2-approximation of vertex cover and is also an MIS of the line graph. The line graph of G is the graph G′ whose vertices correspond to edges of G and an edges exists between vertices of G′ if the corresponding edges in G share a vertex. Even though it is not explicitly mentioned, it can be shown that the LCA algorithm for com-puting approximate maximum matching by [KMNT20] outputs a vertex cover as well. Compared to [Gha22], the approach in [KMNT20] incurs a higher approximation ratio, although only linear dependence on ∆ in the query complexity. 

Local simulation of random order greedy. Let ∆ be the largest set size and f the highest element frequency. Nguyen and Onak [NO08] designed a sublinear-time algorithm to approximate set cover by simulating a classical greedy algorithm on randomly ordered sets. Their algorithm uses 8

2(∆ f )4O(2 ∆)

queries. This complexity was substantially improved in a seminal work by Yoshida, Yamamoto, and Ito [YYI09], who demonstrated how to solve the same problem by (∆ f )O(∆) queries via average case analysis of the randomized greedy MIS algorithm. We note that their output provides an approximation guarantee of H∆ · OPT + εn where n is the number of elements and 

H∆ denotes the harmonic number. For e.g., setting ε = 1 /∆, one gets an H∆ + 1 multiplicative approximation. One can also modify their algorithm to obtain O(log ∆) approximation within

Set cover in O(log n) LOCAL rounds. It is known how to compute approximate set cover in 

O(log n) LOCAL rounds [KMW16, CL23]. Nevertheless, it is unclear how to leverage those algo-rithms in designing LCAs. In particular, 2O(log n) = poly (n); hence, the known techniques would not directly lead to sublinear algorithms. 

# 2 Preliminaries 

2.1 Set Cover and Local Computation Algorithms 

Definition 2.1 (SetCover ). Given a universe of elements E and a family of subsets of E, S ⊆ 2E ,we call C ⊆ S a cover of S if for every e ∈ E there exists at least one S ∈ S containing e. S is said to be optimal if it has the minimum size. We denote the set cover instance as ⟨S , E⟩ and the size of the optimal set cover by OPT (⟨S , E⟩ ) or simply OPT when the instance is clear in context. 

Definition 2.2 (Local Computation Algorithm ( LCA )) . Given a problem Π with input I and desired output O, both presumed to be of large size and ordered, a local computation algorithm for Π is a procedure P that takes as input, (i) an oracle that provides access to any element of I upon query, (ii) an index i with 1 ≤ i ≤ | O|,(iii) oracle access to a sufficiently large common random string, and outputs O[i], i.e., the element of O indexed by i. The query complexity of P is the size of the portion of I that it obtains from the input oracles to obtain O[i].

LCA conventions. In this paper, our goal is to provide an LCA for the Set Cover problem. We shall assume that all set cover instances are represented as a bipartite graph whose one partition denotes the sets and the other denotes the elements. An edge exists in the graph between a set 

S and element e if and only if e ∈ S. We shall assume that this graph is stored in adjacency list representation and the LCA oracles we design can access these lists through input oracles. In particular, the LCA can provide a vertex v in the graph and an integer i to the input oracle which then returns the vertex w that is stored in the ith position of the adjacency list of v. The LCA must support the following queries on a cover C:

• Given a set S, does S belong to C?

• Given an element e, output a set S such that S ∋ e and S ∈ C .9The LCA must support parallel queries, i.e., when the LCA is executed independently on all the sets, the output should be consistent with a given solution to the set cover instance. Additionally, in our algorithms, the randomness can be presampled. That is, given a large common random string, the 

LCA executes a deterministic process. Hence the query complexity is completely deterministic, but the guarantee on the approximation ratio is in expectation over the randomness.

3.1 (Approximate) LCA Simulation of Algorithm 2 

In this section, we design a LCA that almost perfectly simulates Algorithm 2 while using much fewer queries than a naive simulation. The key component of our LCA is the design of two LCA oracles 

IsSetDense () and IsEleCov () that perform probabilistic tests of whether a set S (resp. element 

e) is dense (resp. covered) during (resp. at the end of) iteration (i, j ). See Algorithms 5 and 6. 

High-level description of oracles. The IsSetDense () oracle takes as input (i) an iteration 

(i, j ) and (ii) a set S, then performs a probabilistic test of whether the set S was dense at the beginning of iteration (i, j ). Similarly, the IsEleCov () oracle takes as input (i) an iteration (i, j )

and an element e, then performs a probabilistic test of whether e is covered at the end of iteration 11 (i, j ). These oracles recursively invoke each other as follows. IsSetDense () queries a subset of elements in S and invokes IsEleCov () on these elements at previous iterations to test whether S

is dense. Details of which elements are selected and at what iterations they are tested are discussed later. Similarly, IsEleCov (i, j, e ) queries a subset of the sets S containing e and then invokes 

IsSetDense () on these sets at iterations on or before (i, j ). The details of how the sets are chosen and the iterations in which they are tested are discussed later. Similar oracles are constructed in [GMRV20]. The key difference is our observation that several recursive invocations can be pruned. Using the oracles IsSetDense () and IsEleCov () we can simulate Algorithm 2. We design two LCA oracles GetWeight () and LCA-Weight () . GetWeight () takes as input a set S and returns an estimate of the weight wt (S) assigned by Algorithm 2 to S. The estimate is such that for every set S, the probability that wt (S) computed by the two algorithms differs is o(1 /∆f ). Due to this, it is not guaranteed that wt (S) returned is a valid cover. The oracle LCA-Weight () further modifies the weights so that wt is a valid cover. 

Description of GetWeight (S) and LCA-Weight (S) (Algorithm 3). The goal of proce-dure GetWeight (S) is to compute the fractional weight of a given set S after a probabilistic simula-tion of Algorithm 2. As in Algorithm 2, GetWeight (S) runs iterations (1 , 1) through (log ∆ , log f ).In iteration (i, j ), if S is dense enough in that iteration and wt (S) ≤ 1, GetWeight (S) increases

Analogous to the previous algo-rithm, we design two oracles, weightEstimate and degreeEstimate in Algorithms 9 and 10 that estimate the weights of elements and effective degree of sets at some given iteration (i, j ) after the retro-active updates until phase i⋆. These oracles recursively call each other. To obtain an LCA 

for the weight of a set, one should simulate Algorithm 8 using the oracle (Algorithm 10) designed in this section. They are designed such that they satisfy Eq. (12), Eq. (11) and Eq. (13). We now explain the algorithms in more detail. 28 5.1 Sampling strategy and tracking randomness 

In this subsection, we briefly overview our sampling procedure and how we shall aim to achieve Eq. (11) and Eq. (12). During an iteration (i⋆, j ⋆), to determine if d(S, i ⋆, j ⋆) ≥ ∆/2i⋆

, we make 

2i⋆

· K random samples to S. Based on the pruning strategy of Section 3, we should first remove the elements e ∈ S whose estimated weight after phase 1 is large. If we simply use the LCA 

oracles designed for estimating phase 1, our estimate would not be accurate enough. Firstly, the success probability is not large enough for our guarantee. Secondly, our estimate would then not be independent of the randomness in previous phases. To fix this, observe that for phase i, we are allowed up to 2O(i) queries, thus we may re-execute the oracles for previous phases multiple times, i.e., phase i′ can be re-executed 2O(i−i′) times. We take advantage of this as follows. Before beginning iteration (i⋆, j ⋆) we go over each previous iteration 

(i, j ) ≤ (i⋆, j ⋆) and strengthen the weight estimates of elements after this iteration. Then we use these strengthened estimates for the weights to compute strengthened estimates of the degrees. 

Weight estimates For an element e, we pay for any errors in estimating its weight, purely by the random sampling of sets S ∋ e that contain it. Our guarantees for the estimation of the weights of elements during iteration (i⋆, j ⋆) will depend purely on the the local random bits allocated for e

during (i⋆, j ⋆).

Degree estimates During iteration (i⋆, j ⋆), to estimate d(S) for a set S, we first sample a subset of 2i⋆

· K elements. If these elements capture too many or not enough elements e that were covered during previous iterations, then we say that the estimate failed. If the sampling is sufficient, then the error in estimating the weights of these elements is then based only on an independent sample of sets that contain it, which from the previous paragraph, is independent of any element samples. 5.2 High level overview of oracles 

Below we give an overview of our LCA oracles. 

Oracle degreeEstimate (Algorithm 10). This LCA oracle provides an estimate of the effec-tive degree of set S at the beginning of iteration (i, j ). The oracle returns a probabilistic estimate of the degree of the set S. The guarantees about this estimate are tuned by another set of parameters 

(i⋆, j ⋆). The estimates are returned to the precision needed at iteration (i⋆, j ⋆). We shall later describe how the guarantees vary with each iteration. Intuitively, this guarantee strengthens with every passing iteration in such a way that the number of samples needed increases by at most a 

constant factor with each iteration. 

Oracle weightEstimate (Algorithm 9). This LCA oracle provides an estimate of the weight of an element e at the end of iteration (i, j ). The oracle returns a probabilistic estimate similar to the degreeEstimate oracle. Oracles degreeEstimate and weightEstimate invoke each other. To estimate the degree of a set S at some iteration, we sample elements of S and invoke the weightEstimate oracle to estimate their weight at the end of the previous iteration. To estimate the degree of a set S at the end of iteration (i, j ) after processing up to iteration (i⋆, j ⋆) of the algorithm, we sample 2i⋆

· K

elements e ∈ S. This should be interpreted as strengthening the guarantees by a factor of 2i⋆−i.We shall then prune out the elements e ∈ S that are covered before iteration (i, j ) smoothly by 29 beginning with the first iteration. If we see too many elements staying uncovered, then we give up on the estimate- this is a bad sampling, and our analysis should account for any such errors made. Once i − 1 phases are simulated, we shall have a small enough sample of the set S wherein we can simulate the ith phase. Yet again, we incrementally prune out the elements covered in the iterations 

j′ = 1 , 2, . . . j − 1. The tests are performed such that the probability that an element is estimated to be covered in iteration j′ < j but not in iteration j is 2−Ω( j−j′). This is achieved by boosting the number of samples in the estimate of j′ by a factor of 2j−j′

We can also afford to pretend that most of the weight that covered the given element e came during the last iteration. The weightEstimate oracle on an element e estimates how many sets S that contain e were dense during a given iteration (i, j ). If we find that many sets were dense, then we can estimate that the weight added to that element during this iteration was high enough that it will be covered. # 6 Integral Set Cover 

In this section, we provide the missing details to prove Theorem 1.1, i.e., how to convert the fractional algorithms to integral ones. The ideas used for obtaining an integral version follow from the prior work [GMRV20] and our analysis from the preceding sections. Nevertheless, we provide some details for completeness and outline the algorithm performing the rounding. To convert a fractional algorithm into an integral one, the common change is that whenever a weight of w is added to a set S, S is included in the integral set cover with probability w. This randomness can be sampled in advance, i.e., before an algorithm is executed. We will also need to modify the final naive covering step; the modification is natural. For example, if a set S contains an element e that is not covered and S has the least ID among all sets that contain e, then S will be added to the cover. 

6.1 Converting Algorithm 3 into an integral LCA 

Modifying Algorithm 6. To convert Algorithm 3 into an integral LCA, we need one additional change. In the procedure for Algorithm 6, we can no longer afford to say that an element is “covered” if its “weight” exceeds 1, since it might happen that none of the sets were sampled. We will show below that, on expectation, the weight of every element is O(1) when it gets covered. However, this is still insufficient to obtain LCAs with deterministic query complexity, since wt (e) might be large, albeit with small probability. To remedy this, we use the fact that the random sets chosen to be sampled in iteration (i, j ) can be pre-sampled and we can use these pre-samples to prune out redundant density tests. Essentially, if a set S was not sampled during iteration (i, j ), then we do not need to simulate its state in this iteration. Also, with high probability, we can guarantee that

# 7 Proof of Corollary 1.2 

In this section we give the proof of Corollary 1.2. First we sample 100∆ f sets uniformly at random with replacement. For each set S, we execute the LCA Algorithm 9 to compute whether the LCA 

chose S in the cover. Let k denote the number of such sets. We output mk/ (100∆ f ) as the estimate of the size of the cover. The query complexity is deterministically O(∆ O(log f )). We next argue the approximation ratio. First let us assume we fail if the LCA fails to output an O(log ∆) approximate solution with probability at most 1/100 . Next, if the LCA selected k′ sets, then the expected number of them sampled is at least k′ · 100∆ f /m ≥ 100 . Hence, Corollary 2.4, with probability at least 2/3, the output is O(log ∆) -approximate. To improve the space complexity, we first observe that Algorithm 9 can be implemented with just 

O(log ∆ log f ) space. The state of the recursion can be maintained by storing the current recursive call (set ID S, element ID e, and integers i, j, i ⋆, j ⋆) and the current iteration of the for loops that have been executed. To implement the pruning procedure, we only need to store the number of sets/elements that have passed so far. All of these information can be stores in O(poly (log ∆ log f )) 

bits. When the LCA wants to a particular random bit, suppose that it is fed this as input by an oracle. As such, the total number of random bits needed by Algorithm 9 is ∆O(log f ). We can then use the pseudo-random generator of [Nis92] to simulate ∆O(log f ) truly random bits using only 

O(log ∆ log f ) bits. Finally, the output of a set S depends only on the atmost ∆f other sets that share an element with S. Thus, the entire randomness needed by the algorithm can be generated from just O(∆ f poly (log ∆ log f )) truly random bits with only a poly (log ∆ , log f ) loss in the local runtime of the algorithm. 39 Modifying the approach of [YYI09] for O(log ∆) -approximate Set Cover. The oracle of [YYI09] simulates the following procedure: For i = 1 , 2, . . . ∆, sequentially, create a random ordering of the sets and then iterate over the sets in this order and greedily pick any that have size at least ∆ − i + 1 . They then analyze the average case complexity of an LCA that simulates this procedure. The same analysis when modified with the following procedure yields a query complexity of (∆ f )O(log ∆) . The procedures is as follows: For i = 1 , 2, . . . log ∆ , sequentially, create a random ordering of the sets and then iterate over the sets in the random order and greedily pick any set that has size at least ∆/2i. The analysis of [YYI09] follows almost immediately for this version of the algorithm, at the cost of only losing O(log ∆) factor in the approximation ratio.

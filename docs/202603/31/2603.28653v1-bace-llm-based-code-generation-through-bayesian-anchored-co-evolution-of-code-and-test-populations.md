## 摘要
大语言模型（LLMs）在代码生成方面展现出了卓越的能力。虽然交互式反馈循环可以提高性能，但编写有效的测试并非易事。早期的多智能体框架（如 AgentCoder）实现了这一过程的自动化，但依赖生成的测试作为绝对的真值（ground truth）。这种方法是脆弱的：错误的代码经常通过有缺陷或琐碎的测试，而有效的解决方案往往为了满足错误的断言而被降级。为了解决这一局限性，较新的方法在很大程度上放弃了测试生成，转而采用基于示例的规划和推理。然而，我们认为，如果将生成的测试建模为受贝叶斯更新引导的有噪传感器，它们仍然是宝贵的信号。为此，我们引入了 BACE（贝叶斯锚定协同进化），这是一个将代码合成重新表述为贝叶斯协同进化过程的框架。在该框架中，代码和测试种群在置信分布的引导下进行进化，而置信分布则根据有噪的交互证据进行相互更新。通过将搜索锚定在极少数公开示例上，BACE 防止了自我验证循环中典型的协同进化漂移。在 LiveCodeBench v6（2025年3月后）上的广泛评估表明，BACE 在闭源模型和开源小语言模型上均取得了卓越的性能。

## Abstract
Large Language Models (LLMs) have demonstrated impressive capabilities in code generation. While an interactive feedback loop can improve performance, writing effective tests is a non-trivial task. Early multi-agent frameworks, such as AgentCoder, automated this process but relied on generated tests as absolute ground truth. This approach is fragile: incorrect code frequently passes faulty or trivial tests, while valid solutions are often degraded to satisfy incorrect assertions. Addressing this limitation, newer methods have largely abandoned test generation in favor of planning and reasoning based on examples. We argue, however, that generated tests remain a valuable signal if we model them as noisy sensors guided by bayesian updates. To this end, we introduce BACE (Bayesian Anchored Co-Evolution), a framework that reformulates synthesis as a Bayesian co-evolutionary process where code and test populations are evolved, guided by belief distributions that are reciprocally updated based on noisy interaction evidence. By anchoring this search on minimal public examples, BACE prevents the co-evolutionary drift typical of self-validating loops. Extensive evaluations on LiveCodeBench v6 (post-March 2025) reveal that BACE achieves superior performance across both proprietary models and open-weight small language models.

### 论文总结：BACE —— 基于贝叶斯锚定协同进化的代码生成框架

#### 1. 核心问题与整体含义
论文针对大语言模型（LLM）在代码生成中的**验证脆弱性**问题展开研究。虽然现有的“生成-测试-修复”闭环框架（如 AgentCoder）能提升性能，但它们将 LLM 生成的测试视为绝对真理（Ground Truth）。这导致了两种失效模式：**误报（False Positive）**，即错误代码通过了错误的测试；以及**漏报（False Negative）**，即正确代码因错误的测试断言而被强行修改。
**BACE** 的核心意义在于：它不再盲目信任生成的测试，而是将其视为**有噪声的传感器**，通过贝叶斯概率框架动态评估代码和测试的置信度，从而在不确定的环境中合成高质量代码。

#### 2. 方法论
BACE 将代码合成建模为一个**贝叶斯协同进化过程**，核心组件包括：
*   **概率建模**：为每个代码候选者 $c_i$ 和测试用例 $t_j$ 分别维护一个置信度分布 $b(c_i)$ 和 $b(t_j)$。
*   **有噪传感器模型**：引入超参数来量化不确定性，包括误通过率（$\alpha$）、意外通过率（$\beta$）和巧合通过率（$\gamma$）。执行结果（通过/失败）被视为证据，用于在对数几率（log-odds）空间内更新置信度。
*   **锚定机制（Anchoring）**：利用题目说明中给出的 1-3 个公开示例作为“锚点”（置信度接近 1）。所有代码必须先通过锚点测试才能获得基础信用，这有效防止了协同进化过程中的“自我验证漂移”。
*   **协同进化流程**：
    1.  **初始化**：利用 LLM 生成初始代码和测试种群。
    2.  **交替进化**：偶数代进化测试种群，奇数代进化代码种群，以稳定学习信号。
    3.  **变异算子**：使用 LLM 执行语义交叉、调试修复、重构、差异发现（Differential Testing）等操作。
*   **多样性保留**：通过“行为向量”（代码在所有测试上的通过/失败序列）进行聚类，确保保留具有独特逻辑的个体，而非仅仅保留高分个体。

#### 3. 实验设计
*   **数据集**：使用 **LiveCodeBench v6**（2025 年 3 月之后发布的题目），包含 80 道题目（37 难、25 中、18 易）。选择该数据集是为了确保题目不在 LLM 的训练集中，避免数据污染。
*   **对比方法（Baselines）**：
    *   **Direct Prompting**：直接提示词生成。
    *   **AgentCoder**：经典的基于测试反馈的多智能体框架。
    *   **MapCoder / CodeSIM**：目前最先进的基于规划和模拟的多智能体框架。
*   **评估模型**：闭源模型 **GPT-5-Mini** 和开源小模型 **Qwen2.5-Coder-7b**。

#### 5. 实验数量与充分性
*   **实验规模**：涵盖了不同难度等级的题目，并在两种不同架构（闭源大模型 vs 开源小模型）上进行了验证。
*   **消融实验**：作者针对“锚定机制”进行了专门的消融研究。结果显示，在没有锚点的情况下，困难题目的性能从 51.4% 骤降至 37.8%，证明了该机制的必要性。

#### 6. 主要结论与发现
*   **性能领先**：BACE 在 GPT-5-Mini 上达到了 71.3% 的总 Pass@1，超过了 CodeSIM（67.5%）；在 Qwen2.5-Coder-7b 上表现更优，尤其在中等难度题目上比 CodeSIM 高出 12%。
*   **测试仍有价值**：证明了即使生成的测试不完全准确，只要通过贝叶斯框架合理建模噪声，它们依然能提供强大的学习信号。
*   **多样性是关键**：简单的“优胜劣汰”会导致种群收敛到平庸的错误解，基于行为向量的精英保留策略对进化成功至关重要。

#### 7. 优点
*   **鲁棒性强**：通过贝叶斯更新解决了 LLM 幻觉导致的“错误测试误导”问题。
*   **理论完备**：将进化计算与贝叶斯推理结合，为代码合成提供了更坚实的数学基础。
*   **通用性**：不仅适用于顶级闭源模型，也能显著提升 7B 规模开源模型的代码能力。
*   **差异化测试**：引入差异发现算子来拆分功能等价组，提高了搜索效率。

#### 8. 不足与局限
*   **计算开销**：相比单次推理，协同进化涉及多次 LLM 调用和代码执行，推理成本和延迟较高。
*   **依赖锚点**：如果题目没有提供任何公开示例（虽然罕见），锚定机制将失效，系统可能陷入漂移。
*   **超参数敏感性**：贝叶斯模型中的噪声参数（$\alpha, \beta, \gamma$）可能需要针对不同模型进行微调。
*   **实验样本量**：虽然 LiveCodeBench v6 质量高，但 80 道题目的样本量在统计学上仍有进一步扩大的空间。

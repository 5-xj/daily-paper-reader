## 摘要
在LLMs的最新进展和已有编程演示工作的基础上，终端用户现在可以通过向LLMs明确演示其行为来创建自动化。然而，这些方法依赖于用户知道要自动化什么以及什么能够被自动化的假设。此外，与程序相比，通过LLM代理进行自动化通常成本较高。我们介绍了Motif，一个系统，它被动地观察日常浏览器活动，以发现可编程的重复交互模式，一旦发现模式就向用户提出建议，并在用户确认后生成程序进行安装。用户可以使用自然语言审查和优化程序。我们在一次为期多天的研究中评估了Motif，将其环境发现与用户试图通过“氛围编码”构建的自动化进行了比较。在八名参与者中，Motif发现了比用户自己意识到的更多的可自动化模式。其中大部分与参与者的日常流程相匹配且有用。后续调查显示，大多数人会继续使用Motif生成的程序。

## Abstract
Recent advances in LLMs and existing work on programming by demonstration have made it possible for end users to create automations by explicitly demonstrating their behavior to LLMs. However, these approaches rely on the assumption that users know what to automate and what is capable of being automated. Additionally, automation via LLM agents is often expensive compared with programs. We introduce Motif, a system that passively observes everyday browser activity to discover recurring interaction patterns that are programmable, makes recommendations to users whenever a pattern is discovered and generate a program to install after user confirmation. Users can review, and refine the program using natural language. We evaluated Motif in a multi-day study, comparing its ambient discoveries against automations users attempted to build via ``vibe coding.'' With eight participants, Motif discovered more automatable patterns than users recognized. Most of them matched participants' routines and were useful. Follow-up surveys showed most would continue using Motif-generated programs.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：现有终端用户自动化方法（如编程演示PBD、LLM代码生成如“氛围编码”）都假设用户必须先知道“要自动化什么”并主动发起过程。但实际上，用户存在“启动鸿沟”（initiation gap）：任务习惯化导致重复操作不可见、缺乏可自动化任务的思维模型、难以精确描述需求、以及自动化前期投入成本高。因此，**用户往往无法识别哪些日常浏览器操作可以被自动化**。
- **整体含义**：本文提出**环境自动化发现（ambient automation discovery）**的新范式——系统被动观察用户日常行为，自动发现可编程的重复交互模式，并将其转化为低成本、确定性的程序，用户只需审查和确认即可部署。这从根本上降低了自动化启动门槛，使更多用户受益。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- 被动监测浏览器活动，利用LLM挖掘语义丰富的重复模式，生成个性化、可执行且成本低廉的Tampermonkey脚本，而不是使用昂贵的LLM代理（agent）。

### 关键技术细节（流程）

1. **数据采集（Data Collection）**：收集四类日志：
   - UI交互（点击、复制粘贴等，附带周围文本和ARIA标签）
   - 浏览器系统交互（标签切换、URL）
   - OS级交互（Chrome窗口截图、鼠标绝对位置、离开/返回时间）
   - 轻量网络交互（UI交互后1.5秒内的API请求和响应）

2. **动作抽象（Action Abstraction）**：利用LLM多模态能力，将原始交互日志转化为自然语言描述的高层动作，附带元数据（如复制文本、页面URL）。动作被存入数据库形成序列。

3. **模式挖掘（Pattern Mining）**：LLM根据动作序列识别可自动化的重复模式，与新批次数据合并相似模式。使用工具（代码解释器、URL爬取、Google搜索）辅助理解上下文。阈值为模式出现次数≥5次才显示。

4. **程序设计（Program Design）**：LLM为每个模式生成用户友好的设计，包含：
   - **触发条件**：按钮、快捷键或自动触发
   - **程序描述**：非技术语言描述行为
   - **通知机制**：何时通知用户（成功/失败等）

5. **程序排序（Program Ranking）**：LLM根据频率、未来发生可能性计算有用性分数，从高到低排序。

6. **用户审查（User Review）**：用户可通过界面查看、编辑设计（自然语言修改），或点击“重新生成设计”。

7. **程序生成（Program Generation）**：用户确认后，LLM生成Tampermonkey脚本，包含明确的@grant作用域声明，以确保安全。

### 关键设计目标（DG1~DG4）
- DG1：无需用户发起发现
- DG2：基于实际行为生成个性化规格
- DG3：用户保持控制（审查、修改、拒绝）
- DG4：生成确定性、低成本、可复用的程序

## 6. 论文的主要结论与发现

- **自动发现的有效性**（RQ1）：Motif平均发现22个可自动化模式，而用户自我反思仅平均1.3个。参与者在审查的40个模式中，85%（34/40）匹配日常流程，92%（37/40）被认为有用（若程序完美运行）。
- **程序生成质量**（RQ2）：60%的程序按预期运行（24/40）。低难度程序成功率82%，中难度45%，高难度0%。自动触发和快捷键触发的程序成功率更高。
- **用户感知**（RQ4）：Motif在易用性、理解代码、验证正确性、乐趣方面得分显著高于氛围编码。6/8参与者表示愿意继续使用Motif。
- **长期实用性**（RQ5）：纵向调查中，30%程序被定期使用，40%使用过几次；大多数参与者认为程序按预期工作、节省时间，并计划继续使用。
- **改变用户认知**：所有参与者同意使用Motif后改变了对浏览器工作流的看法，但自新识别任务的能力仍有限（仅1个有效模式）。

- **新颖范式**：首次提出“环境自动化发现”，解决了传统方法中用户必须主动识别的瓶颈，真正做到了“系统为你找出可自动化的事情”。
- **技术融合**：结合了被动数据采集、LLM语义模式挖掘、程序生成和交互式审查，形成完整闭环。

## 8. 不足与局限

- **实验覆盖**：
  - 样本量仅8人，且均为学生，可能无法代表更广泛的职场用户或非英语用户。

  - 实验室会议仅审查5个程序/人，可能遗漏有价值的更高位模式。
- **偏差风险**：
  - 自选基调，参与者对Motif可能有正向偏见。
  - 隐私担忧导致部分用户仅在小范围任务中使用Motif，可能低估模式多样性。
- **技术局限**：
  - 生成的程序受限于Tampermonkey的页面内运行能力，无法处理浏览器外部交互（如系统弹出窗口）。

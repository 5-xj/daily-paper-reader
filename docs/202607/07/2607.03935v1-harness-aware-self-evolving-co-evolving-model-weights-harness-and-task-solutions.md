## 摘要
自进化框架通常优化任务解决方案，同时将周围的框架视为固定的。我们引入了框架感知自进化（HASE），一种智能体强化学习框架，其中单个模型可以在多轮动作空间中生成任务解决方案或编辑选定的框架组件。HASE使得单个Qwen3-8B模型能够匹配使用Claude Code作为框架提供者的GPT-OSS-120B模型的文本分类性能。在alpha因子挖掘中，HASE优于已报道的GPT-OSS-120B基线。HASE还修复了不完善的评估组件，并在圆填充算法发现中收敛到最先进的性能。这些结果表明，HASE通过一个统一的智能体过程改进了框架和解决方案。

## Abstract
Self-evolving frameworks usually optimize task solutions while treating the surrounding harness as fixed. We introduce Harness-Aware Self-Evolving (HASE), an agentic reinforcement-learning framework in which a single model can generate task solutions or edit selected harness components in a multi-turn action space. HASE enables a single Qwen3-8B model to match the text-classification performance of a GPT-OSS-120B model that uses Claude Code as the harness proposer. In alpha factor mining, HASE outperforms the reported GPT-OSS-120B baseline. HASE also repairs imperfect evaluation components and converges to state-of-the-art performance in circle-packing algorithm discovery. These results show that HASE improves the harness and the solution through one unified agentic process.

## 论文总结：Harness-Aware Self-Evolving (HASE)

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有的基于大语言模型（LLM）的自我进化系统，如FunSearch、AlphaEvolve或基于测试时强化学习的系统，通常将“框架（Harness）”视为固定的，只优化“对象级解决方案”。这里的“框架”指的是所有中介组件，包括提示词、记忆、上下文检索、工具接口、状态管理器和评估器等。这种固定假设有两个主要缺点：
    1.  **评估器不完善**：人工编写的评估器可能不完整或与外部环境不一致，导致模型在分数驱动下可能“钻空子”（Reward Hacking），而不是真正解决问题。
    2.  **引导组件不佳**：引导性的框架组件（如提示词）可能提供较差的搜索先验，限制了模型发现更好解决方案的能力。
- **整体含义**：本文提出，模型不应只在固定的环境中求解，而应能够主动优化其工作环境本身。通过同时进化模型权重、任务解决方案和框架组件，可以显著提升系统整体性能，使较小的模型能匹敌或超越依赖强大外部工具的大模型系统。

### 2. 论文提出的方法论

- **核心思想**：提出**Harness-Aware Self-Evolving (HASE)**，一个统一的智能体强化学习框架。该框架允许**同一个模型**在**多轮动作空间**中，既可以生成任务解决方案，也可以编辑选定的框架组件。HASE将框架的改进与解决方案的生成融为一体。

- **关键技术细节**：
    1.  **框架分类**：HASE区分两种可编辑的框架组件：
        - **引导组件（Guidance Components）**：包括提示词、记忆格式、检索策略等，负责改进搜索过程。
        - **评估组件（Evaluation Components）**：包括验证器和评分器，定义解决方案的有效性和任务指标。
    2.  **统一动作空间**：模型可以执行两类动作：
        - `提交解决方案`：提交一个对象级代码或结果。
        - `编辑框架`：使用沙盒命令（如`ls`, `cat`, `replace`）来检查和修改可编辑的框架文件。
    3.  **训练循环（HASE算法）**：
        - **内部循环（GRPO Epoch）**：模型在当前的框架下进行多轮轨迹（Trajectory）的采样。每个轨迹可以包含解决方案提案和框架编辑。编辑后的框架仅在该轨迹内临时生效。奖励只基于最终提交的解决方案质量（任务分数）。
        - **相位级别审查（Phase-level Review）**：在每个训练相位结束后，系统会审查该相位内所有候选的框架编辑（Harness Edits）。
            - **引导组件审查**：通过轻量级的概念验证（Proof-of-Concept）分数（即编辑后解决方案分数是否提升）来筛选候选，并在验证集上评估，只提交能提升验证分数的编辑。
            - **评估组件审查**：只有当本地评估器（Local Evaluator）的代理分数与外部真实评估器（Real-world Evaluator）的真实分数出现偏差时，才允许进行修复。被修复的评估器必须能解决偏差集合（`mismatch set M`）中的案例，同时不破坏已有功能。这防止了无限制的奖励劫持。
    4.  **奖励设计**：HASE不奖励简单的框架访问或随机修改动作。奖励只来源于最终提交的解决方案的任务性能。框架编辑通过提升解决方案质量来间接获得奖励。

### 3. 实验设计

- **数据集与场景**：在四个控制力不同的任务上进行了评估：
    1.  **文本分类**：使用Symptom2Disease (S2D)数据集，专注于**引导组件**的进化。基准来自Meta-Harness。
    2.  **Alpha因子挖掘**：使用中国A股CSI300指数的量化金融数据（CogAlpha基准），验证**引导组件**进化。目标是从历史市场数据中生成预测未来收益的信号。
    3.  **圆填充（Circle Packing）**：在单位正方形内放置不相交的圆，目标是最大化总面积。被设计为**评估组件**修复的挑战性任务。
    4.  **海尔布隆三角形（Heilbronn Triangle）**：在单位等边三角形内放置N个点，使任意三点构成的最小三角形面积最大化。同样用于测试**评估组件**的自修复能力。

- **基准与方法对比**：
  - 文本分类：与Zero/Few-shot GPT-OSS-120B、ACE、MCE、Meta-Harness (GPT-OSS-120B + Claude Code) 对比。
  - Alpha因子挖掘：与Llama3 8B/70B、GPT-OSS-20B/120B、GPT-4.1、o3等方法对比，指标为RankIC、AER、IR等。
  - 圆填充：与AlphaEvolve的SOTA（State-of-the-art）分数对比。
  - 海尔布隆三角形：与固定模型权重的“冻结”基线（FREEZE）进行对比消融实验。

### 6. 论文的主要结论与发现

- HASE能够有效地协同进化模型权重、框架和任务解决方案。

- 在算法发现任务中，HASE能够**自主发现并修复不完美的评估器**，通过“偏差驱动修复”机制，逐步完善评估器，最终达到或超越SOTA性能。
- 模型权重的共同进化（Co-Evolution）是必要的：仅仅修复评估器（固定模型权重）会导致性能停滞，而HASE的权重更新可以让策略内化约束，实现持续优化。

### 7. 优点

- **方法创新性**：将框架视为可进化的一部分，并用一个统一的智能体过程来同时求解和编辑框架，这是一个新颖且有效的思路。
- **安全性设计**：通过严格的相级审查、将评估器编辑与真实世界反馈锚定、引入“偏差集合”等机制，有效地防止了无限制的奖励劫持，保证了评估器进化的安全性。
- **高效的资源利用**：证明了一个小得多的模型（8B）通过自身的进化，可以达到或超过依赖昂贵的外部大模型作为“框架提议者”的复杂流水线系统的性能。这降低了对超大模型的依赖。
- **实验严谨性**：在Alpha因子挖掘实验中，进行了极其严谨的未来信息泄漏检测和多层审计，大大增强了结果的可靠性。海尔布隆的消融实验也干净利落地证明了共同进化的价值。
- **清晰的任务覆盖**：设计了多种任务来分别测试和验证引导组件与评估组件的进化能力，论证非常全面。

### 8. 不足与局限

- **非穷尽的缩放研究**：论文承认，其实验不是大范围的缩放研究，每个领域还有更困难的后续设置。验证主要集中在8B模型上，未深入研究HASE在不同规模模型（如70B）上的扩展性。
- **保守的工程选择**：HASE为了安全性（Safety），做出了一些保守选择，例如：只允许编辑白名单内的框架组件、评估器编辑需要外部验证（真实反馈的偏差）、相级审查严格。这些限制可能会拒绝一些有用但更激进的修改，从而牺牲了部分通用性来换取安全性。
- **引导组件编辑的表面局限性**：模型对引导组件的编辑仅限于文件读写，没有更复杂的API调用或代码重构能力，这可能限制了其进化潜力。
- **“偏差集合”的局限性**：评估器的修复依赖于已发现的偏差案例，可能无法一次性发现所有潜在的评估器缺陷，需要持续暴露问题才能逐步修复，在更复杂的场景下可能效率不高。
- **对真实世界反馈的依赖**：评估器修复依赖于查询昂贵或稀疏的真实世界反馈（如排行榜或模拟器），这可能在实际部署中构成瓶颈。

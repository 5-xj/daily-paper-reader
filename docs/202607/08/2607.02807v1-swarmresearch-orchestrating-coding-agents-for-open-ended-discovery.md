## 摘要
长期运行的编码智能体（如自动研究型智能体）能够持续发现开放式问题的优化方案。然而，它们往往收敛到单一的高级方法，然后进行低级编辑，而忽略了其他更优的解决方案。我们假设两种框架层面的设计选择导致了这种行为：在单个长期运行的智能体中积累上下文，以及只暴露单一的程序状态进行编辑。我们引入了SwarmResearch，一种协调器-子智能体框架，其中Shepherd智能体使用全局上下文来引导一组搜索智能体，每个搜索智能体在其各自的git分支中操作局部上下文。在开放式优化任务中，SwarmResearch在13/15个任务上发现了比最先进的LLM引导进化和多智能体技术更好或相当的解，这得益于更高级别的探索。与固定规模的串行和并行智能体相比，SwarmResearch的协调器引导扩展通过在不同搜索深度自适应地调整并行度，发现了性能更优的解。

## Abstract
Long-running coding agents such as autoresearch can persistently discover optimizations for open-ended problems. However, they tend to converge onto a single high-level approach, then proceed with low-level edits while missing other superior approaches to the problem. We hypothesize two harness-level design choices contribute to this behavior: accumulating context in a single long-running agent and only exposing a single program state to edit. We introduce SwarmResearch, an orchestrator-subagent harness in which a Shepherd Agent uses global context to steer a population of Search Agents, each operating with local context in their respective git branch. On open-ended optimization tasks, SwarmResearch discovers better or comparable solutions to state-of-the-art LLM-guided evolution and multi-agent techniques on 13/15 tasks, driven by higher-level exploration. Compared with fixed scaling of serial and parallel agents, SwarmResearch's orchestrator-guided scaling discovers better-performing solutions by adapting parallelism at different search depths.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：长期运行的编码智能体（如 AutoResearch）在解决开放式优化问题时，容易收敛到单一的高级方法，随后陷入大量的低级微调（micro-optimization），而忽略了其他可能更优的探索方向。这种“过早收敛”行为导致搜索空间被局限在局部最优。
- **根本原因**：作者假设两个框架层面的设计缺陷：
  - **上下文积累**：单一智能体长时间运行后，对话历史中积累了大量关于当前方法的细化信息，使得转向全新方案的概率大大降低。
  - **单一程序状态**：所有改进都提交到同一个可编辑程序上，不保留竞争性的分支，导致贪心局部搜索。
- **研究意义**：提出一种协调器-子智能体架构，通过解耦全局与局部上下文、利用Git分支保存多版本方案，从而促进更高层次的探索，提升开放式发现的效率和质量。

## 2. 提出的方法论

### 核心思想
SwarmResearch 采用 **协调器（Shepherd Agent）- 子智能体（Search Agents）** 架构：
- **Shepherd Agent**：维护全球上下文（所有搜索智能体的摘要、得分、方向），决策如何分配子智能体。
- **Search Agents**：在各自的Git分支上执行实验，只拥有局部上下文（只读自己分支的内容）。
- 通过**父选择**、**智能体类型（Explorer/Optimizer）**、**提示补充**三种机制进行动态引导。

### 关键技术细节

1. **上下文管理**：
   - **Explorer 智能体**：使用全新的上下文窗口，仅基于工作目录内容（无历史），鼓励全新方案。
   - **Optimizer 智能体**：继承父智能体的对话历史，适合在已有方案上做串行精炼。
   - 每个搜索智能体完成实验后，将简短摘要写入 `findings.md`，该文件仅被同线智能体继承。

2. **Git 管理**：
   - 每个搜索智能体启动时创建新的Git分支和工作目录。
   - 分支可来自 `main`（基线）或已有智能体的提交，支持扇出（并行探索）和串行继承。
   - 多父合并（git merge）允许组合不同方案。

3. **Shepherd 的决策流程**：
   - 初始化阶段：从 baseline 扇出大量 Explorer 智能体，确保多样性。
   - 探索阶段：在有前景的分支上启动序列化的 Explorer/Optimizer。
   - 遇到平台期：重新从 baseline 启动 Explorer，并提示避免已探索方向。
   - 重要限制：Shepherd 不得指定具体想法，只提供上下文和约束。

4. **代码实现**：以三个草稿技能（Shepherd、Explorer、Optimizer）形式实现，兼容 Claude Code 等工具。

## 3. 实验设计

### 基准（Benchmark）
包含三个领域、15个开放式优化任务：
- **数学优化**（5个）：Circle Packing、Signal Processing、Erdos Min Overlap、MMD-14-3、3rd-Autocorrelation。
- **系统优化**（5个）：EPLB、LLM-SQL、Txn Scheduling、Cloudcast、PRISM（来自 ADRS 基准）。

- **固定规模扩展**：并行+串行迭代（n,k）组合（5,12;10,6;15,4;20,3;30,2），使用 Minimax-M2.5 和 Pi 框架，总迭代60次，取最优配置。

## 6. 主要结论与发现

1. **性能优势**：SwarmResearch 在13/15个任务上超过或媲美EvoX和CORAL。特别在启发式任务上优势明显（如AHC016提升约20%）。

3. **协调器引导扩展超过最优固定扩展**：在4/5个任务上，SwarmResearch自适应并行深度优于任何固定(n,k)配置。

5. **Shepherd行为特征**：默认倾向贪心搜索（收敛到top方案），但通过提示可鼓励保持多样性。

## 7. 优点

- **设计简单有效**：仅通过上下文管理和Git分支两个核心改动，显著提升探索多样性，无需复杂超参数。
- **兼容性好**：以三个技能（skills）实现，可直接集成到现有编码智能体工具（如Claude Code）中。
- **动态扩展**：协调器自动调整并行/串行比例，优于手动预设的固定扩展策略。
- **细粒度分析**：通过代码变更粒度验证了高层次探索增加的论断。
- **案例研究深入**：展示了实际应用场景的完整搜索轨迹、最佳方案以及失败案例，提供了可重复性细节（如附录中的技能代码）。

## 8. 不足与局限

- **LLM创造力限制**：SwarmResearch 不增强基础模型的创造力，探索的仍是已知方法的变体，并非真正的全新突破。
- **Shepherd行为依赖提示**：当前大型语言模型（如Opus 4.6）并不能很好地执行复杂的种群管理（如合成瓶颈、分配差异化任务），需要大量人工提示工程。
- **单次运行**：由于成本高昂，主要实验仅运行一次，统计稳定性不足。作者也承认方差问题，并建议未来需研究如何分配预算实现最佳重现性。
- **模型依赖性**：EvoX在使用GPT模型时性能更好，但报告中使用Opus 4.6可能低估了其表现。CORAL的作者报告结果也高于本研究复现结果。

- **版本管理开销**：虽然Git分支有助于保持多样性，但也增加了被放弃分支带来的计算浪费。

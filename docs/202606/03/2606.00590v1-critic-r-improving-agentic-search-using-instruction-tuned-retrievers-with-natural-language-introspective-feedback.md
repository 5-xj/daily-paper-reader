## 摘要
智能体搜索系统通过迭代地与检索模型交互来回答复杂查询。尽管取得了实质性进展，但优化用于智能体搜索的检索器仍然具有挑战性，通常需要大量的协同训练或金标准标注，这限制了其在实际中的应用。我们提出Critic-R，这是一个在推理和训练过程中明确闭合推理智能体与检索模型之间反馈循环的框架。Critic-R引入了一个批评模型，在消费检索到的证据后评估智能体的内省推理轨迹，以确定检索到的上下文是否足以支持下一步推理。Critic-R包含两种互补机制：Critic-R-Zero，一个推理时的查询细化循环，迭代地重写查询和检索指令；以及Critic-Embed，一种检索模型的优化方法，利用成功和失败的精化轨迹作为自动监督，无需手动相关性标注。我们在HotpotQA、2WikiMultihopQA、MuSiQue和Bamboogle上评估Critic-R。结果表明，Critic-R显著提高了检索质量和下游答案准确性。

## Abstract
Agentic search systems iteratively interact with retrieval models to answer complex queries. Despite substantial progress, optimizing retrievers for agentic search remains challenging, often requiring heavy co-training or gold-standard annotations that limit real-world applicability. We propose Critic-R, a framework that explicitly closes the feedback loop between the reasoning agent and the retrieval model during both inference and training. Critic-R introduces a critic model that evaluates the agent's introspective reasoning trace after consuming retrieved evidence to determine whether the retrieved context sufficiently supports the next reasoning step. Critic-R has two complementary mechanisms: Critic-R-Zero, an inference-time query refinement loop that iteratively rewrites queries and retrieval instructions, and Critic-Embed, an optimization approach for retrieval models that leverages successful and failed refinement trajectories as automatic supervision without requiring manual relevance annotation. We evaluate Critic-R on HotpotQA, 2WikiMultihopQA, MuSiQue, and Bamboogle. Results show that Critic-R significantly improves both retrieval quality and downstream answer accuracy.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：现有智能体搜索系统中，检索器通常是冻结的黑箱组件，多数方法只优化推理智能体，假设其能通过查询改述来弥补检索失败。然而，检索质量实际上是系统性能的关键瓶颈。
- **现有局限**：优化检索器需要大量协同训练（如Agentic-R、CoSearch）或金标准相关性标注，难以在推理模型不可训练、检索器由外部提供或无标注时部署。
- **研究目标**：在不依赖手工标注的前提下，利用推理智能体的**内省反馈**（反思检索结果是否满足信息需求）对检索器进行自动监督，实现检索质量的闭环优化。
- **核心意义**：提出一个同时作用于推理时和训练时的通用框架，可适配任意推理智能体和检索器，显著提升多跳问答准确性。

## 2. 方法论：核心思想、关键技术细节与算法流程

- **核心思想**：引入一个独立的**批评模型**，在每次检索后通过分析推理智能体的内省推理轨迹（消费检索文档后的下一段思考）来判断检索结果是否充足。若不充足，则生成细化后的查询和检索指令，并标记该次检索为负例；否则标记为正例。
- **两种互补机制**：
  - **Critic-R-Zero（推理时细化循环）**：在推理循环中，对每个搜索动作进行最多K次细化。初始检索后，将结果“投机地”提供给推理智能体生成下一段思考（但不正式提交），批评模型根据该思考判断是否满足需求。若否定则生成新查询和指令重试，直至满意或耗尽预算，最终将满意的文档提交至历史。
  - **Critic-Embed（检索器微调）**：利用Critic-R-Zero在执行训练集问题时产生的轨迹，自动收集正例（满足需求的文档）和轨迹内硬负例（被拒绝的文档）。使用InfoNCE对比损失，结合批次内负例和轨迹内硬负例，微调检索器（基于Stella-400M）。
- **算法流程**（文字描述）：
  1. 推理智能体MR在每个步骤生成思考Ti和动作Ai（搜索或回答）。
  2. 若为搜索，提取初始查询qi^(1)和默认指令I_i^(1)，进入细化循环。
  3. 在循环中，检索器R返回top-k文档D_i^(t)，投机地交给MR生成下一步思考T_i+1^(t)。
  4. 批评模型MC判断是否满意：若满意，将D_i^(t)加入正例集̃D⁺并终止循环；若不满意，加入负例集̃D⁻，并利用批评模型生成新查询和指令，进入下一次细化（t+1）。
  5. 循环结束（最多K次）后，将最终选定的文档提交至推理历史。
  6. 结束后返回最终答案以及收集的正负文档（仅用于训练）。
- **损失函数**：InfoNCE损失，基于查询嵌入、正例嵌入、批次内负例和轨迹内硬负例计算。

## 6. 主要结论与发现

1. **推理时细化有效**：任何规模的批评模型都比无批评基线显著提升性能（平均12.4%相对改善），且较大批评模型（72B）不总是最佳，32B在多跳任务中往往最优。
2. **Critic-Embed优于基线**：仅替换检索器（无推理循环），Critic-Embed即全面超越Stella-400M和Agentic-R，尤其在k=1时优势最大。说明从Critic-R-Zero轨迹中学习的对比训练有效。
3. **组合策略最优**：Critic-R（Critic-Embed + 推理循环）在整体平均上取得最佳结果（EM 0.3957, F1 0.4959），比单独使用循环或训练检索器更好。两者互补。
4. **内省反馈是关键**：移除推理智能体的内省反馈后，Critic-Embed性能显著下降（平均F1下降约0.03），证明了该设计的重要性。
5. **通用QA也有收益**：在NQ、TriviaQA、PopQA上，Critic-R-Zero同样带来持续改善。

## 7. 优点

- **无需手工标注**：完全利用推理智能体的内省反馈自动生成训练信号，降低了人工成本。
- **双循环互补**：推理时细化（处理失败）与训练时优化（减少失败）结合，形成闭环。
- **框架通用**：批评模型和推理智能体可以独立替换，适用于任意ReAct式智能体和任意检索器。
- **实验全面**：覆盖多种模型规模、检索深度和数据集，消融实验严谨。
- **方法创新**：将推理智能体的“思考”作为检索质量的代理信号，设计精巧。

## 8. 不足与局限

- **依赖推理智能体反馈质量**：较弱或未充分训练的推理模型可能无法提供准确的内省轨迹，从而削弱批评模型的判断，进而影响整体性能。
- **语料静态性**：实验仅使用静态Wikipedia语料，未在动态网络搜索、实时信息更新或私有企业文档等噪声更大、分布漂移更显著的环境中评估。
- **训练数据偏差**：Critic-Embed的训练数据仅来自HotpotQA和MuSiQue的训练集，可能对其他领域或分布的任务泛化性有限。
- **算力开销未量化**：Critic-R-Zero增加了多次检索和批评模型推理调用，但论文未具体分析额外延迟和计算成本，实际部署时需权衡。

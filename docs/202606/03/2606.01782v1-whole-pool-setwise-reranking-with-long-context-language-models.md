## 摘要
以往基于大语言模型的段落重排序器往往昂贵且缓慢，因为输入上下文限制要求大语言模型进行大量依赖性的模型调用。我们研究了近期长上下文大语言模型如何改变这一问题：当全部检索到的候选段落可以一次性展示给模型时，排序不再需要从大量重叠的局部比较中重建。我们提出了整池集合式重排序（Whole-Pool Setwise re-ranking），其中每次调用考虑所有当前未排序的候选段落，并引入了DualEnd方法，该方法在一次调用中同时识别出最相关和最不相关的段落。通过从两端填充排序，DualEnd用50次串行大语言模型调用对100个候选进行排序，而可比的一次一个段落的整池方法需要99次调用。我们在两个段落重排序基准上使用九个开放权重的大语言模型进行了实验，衡量了有效性、调用次数、令牌使用量、运行时间和输出可靠性，结果表明长上下文不仅仅是更多的提示空间，更是使大语言模型重排序器既有效又高效的机会。

## Abstract
Previous LLM-based passage re-rankers are often expensive and slow because the input context constraints require the LLM to make many dependent model calls. We study how recent long-context LLMs change this problem: when the full set of retrieved candidate passages can be shown to the model at once, ranking no longer has to be reconstructed from many overlapping local comparisons. We propose Whole-Pool Setwise re-ranking, where each call considers all currently unranked candidate passages, and introduce DualEnd, which identifies both the most and least relevant passages in one call. By filling the ranking from both ends, DualEnd ranks 100 candidates with 50 serial LLM calls, compared with 99 calls for comparable one-passage-at-a-time whole-pool methods. Experiments with nine open-weight LLMs on two passage re-ranking benchmarks, measuring effectiveness, call count, token use, runtime, and output reliability shows that long context is not merely more prompt space, but an opportunity to make LLM re-rankers both effective and efficient.

## 1. 核心问题与整体含义（研究动机和背景）
- **传统基于 LLM 的段落重排序器**：受限于输入上下文长度，每次只能比较局部窗口内的候选段落。为得到全局排序，需要依赖多次串行调用（如冒泡排序、堆排序），导致推理成本高、速度慢。
- **长上下文 LLM 的新机会**：Gemma 3（128k）、Qwen 3.5 和 Gemini 3.1（1M）等模型支持超长上下文，使得一次性将所有候选段落放入单一上下文成为可能。这改变了设计问题：从“如何把候选塞进上下文”变为“如何从每次上下文传递中提取最多的排序信息”。
- **本文目标**：探索长上下文条件下，如何让每个 LLM 调用贡献更多排序决策，在保证有效性的同时大幅减少串行调用次数，实现高效且有效的重排序。

## 2. 方法论：核心思想、关键技术细节、算法流程
- **核心思想**：提出 **Whole-Pool Setwise 重排序**（WP-Setwise），每次 LLM 调用考虑当前所有未排序的候选段落（即整池）。基于此，设计 **DualEnd** 策略，在一次调用中同时识别最相关（最佳）和最不相关（最差）的段落，然后从最终输出排序的两端（头部和尾部）向内填充。
- **关键技术细节**：
  - 三种变体：
    - **WP-Setwise-Top (WP-T)**：每次选择最相关段落，放入头部，调用 N-1 次。
    - **WP-Setwise-Bottom (WP-B)**：每次选择最不相关段落，放入尾部，调用 N-1 次。
    - **WP-Setwise-DualEnd (WP-DE)**：每次同时选择最相关和最不相关段落，分别放入头部和尾部，调用 N/2 次。
  - 输入：查询 q、BM25 初始池 N（候选段落集合）。
  - 输出：完整排序 R，长度为 |N|。
  - 每一步，对当前存活池 N'（未排序段落）进行数值重标号（1…|N'|），LLM 按照指定格式输出两个标签（Best: <数字>, Worst: <数字>）或单一标签（WP-T/WP-B）。解析失败时最多重试 3 次，最终失败则退回到初始排序。
  - 提示模板：使用数值标签（避免字母不够用）、固定格式要求（如“Reply with exactly two distinct passage numbers…”）、零样本指令。
- **算法流程（WP-DE）**：
  1. 初始化：`N' = N`，`R` 为空数组，`head=1, tail=|N|`。
  2. 当 `|N'| ≥ 2`：
      - 重标号 N' 为 1…|N'|。
      - 调用 LLM DualEnd 得到 (imost, ileast)。
      - 将 N'[imost] 放入 R[head]，N'[ileast] 放入 R[tail]。
      - 从 N' 中移除这两个段落，head++，tail--。
  3. 若最后剩一个段落，直接放入 R[head]。
  4. 返回 R。

## 3. 实验设计
- **数据集**：TREC Deep Learning 2019 (DL19) 和 2020 (DL20)，分别有 43 和 54 个有判断的查询。文档集为 MS MARCO（约 8.8M 段落）。
- **基准设置**：使用 Pyserini 提供的 BM25 top-N 池作为初始候选，N ∈ {10, 20, 30, 40, 50, 100}。所有重排序方法基于同一初始池，不添加额外段落。
- **对比方法**：
  - **窗口式 Setwise 基线**（原始方法）：SW-TB（Top Bubblesort）、SW-TH（Top Heapsort）、SW-BB（Bottom Bubblesort）、SW-BH（Bottom Heapsort）。窗口大小固定为 3，只输出 top-10 排序（与原文一致）。
  - **整池式变体**：WP-T、WP-B、WP-DE（均输出完整排序）。
- **评估指标**：nDCG@10、nDCG@N、串行 LLM 调用次数、令牌使用量（prompt/completion）、运行时间、解析失败率。进行配对非劣效性检验（δ=0.01，BH-FDR 校正）。
- **模型**：9 个开放权重指令微调模型：

## 6. 主要结论与发现
- **WP-DE 大幅降低 LLM 调用次数**：对 100 个候选，WP-DE 仅需 50 次串行调用（WP-T/WP-B 需 99 次），窗口式基线 SW-TH 需 123 次，SW-BB 需 2500 次。调用次数减少 49.5% 且是算法性的，非实现细节。
- **有效性接近 SOTA**：WP-DE 平均 nDCG@10 为 0.627，略低于 WP-T（0.645）和 SW-TB（0.673），但在多数模型上差异不显著（仅 4/8 个模型显著高于 WP-DE）。在全排序 nDCG@N 上，WP-DE（0.528）与 WP-T（0.535）和最强 SW 变体（0.534）差距仅 0.007。
- **令牌和运行时间减半**：pool=100 时，WP-DE 平均每查询 254.7K 令牌和 42 秒；WP-T/WP-B 约 501-507K 令牌和 67-91 秒。
- **解析可靠性高**：WP-DE 平均每查询仅 0.066 次解析回退，而 WP-T 为 0.854，WP-B 为 0.300。
- **位置偏差存在性**：反转和打乱池顺序会降低 nDCG@10，仅 Llama3.1-8B WP-DE 反转条件有显著下降。说明整池提示对输入顺序敏感，但并非所有组合均显著。
- **长上下文 LLM 的核心优势**：不仅提供更大提示空间，更可让每个 LLM 调用做出多个排序决策，实现效率与效果的平衡。

## 7. 优点
- **方法创新简洁**：DualEnd 策略简单但高效，利用了长上下文的全局视角，避免传统方法中大量局部比较的重建开销。
- **实验全面**：覆盖多模型规模、多池大小，提供成本-效果可视化（图 2），包括令牌、时间、解析失败等实用指标。
- **稳定性与位置偏差分析**：补充了重复运行和顺序控制实验，提升了结论的可靠性。
- **开源代码**：在 Apache 2.0 许可证下发布，便于复现和扩展。
- **实际价值**：在 LLM 调用预算受限的场景下（如在线系统），WP-DE 能以极低调用次数提供具有竞争力的全排序。

## 8. 不足与局限
- **实验覆盖有限**：仅基于 BM25 初始池，未验证在稠密检索（如 DPR）或混合引擎下的表现；仅使用 TREC DL 2019/2020 两个数据集，泛化性待检验。
- **位置偏差风险**：整池提示对输入顺序敏感，虽进行了控制实验，但未全面探索所有可能的顺序、标签、提示格式效应。提示模板（如数值标签、格式要求）可能影响模型行为。
- **单一解码策略**：使用贪婪解码，未考虑排列自一致性、多采样等校准方法，可能低估或高估效果。
- **未评估超大规模池**：仅测试最大池大小 100，未涉及更大候选集（如 200、1000）。长上下文模型在处理极长上下文时仍有“迷失在中间”等问题，可能影响实际部署。
- **伦理风险**：虽然使用公开数据和模型，但 LLM 生成输出仍可能包含有害、偏见内容，论文对此作了说明。
- **仅关注重排序阶段**：未改善检索召回，仅对固定候选集进行重排序，且使用单轮提示、无强化学习等高级技术。

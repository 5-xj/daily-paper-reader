## 摘要
本文介绍了RACL，一种用于元启发式算法的推理智能体控制层。RACL在现有优化器之上放置一个推理智能体。该智能体不替换优化器，也不修改业务约束。相反，它通过观察运行内存、推理过去行为、制定有界假设、测试干预、评估结果、应用护栏、整合有用策略并解释其决策，来控制优化器的内部搜索行为。实验以车辆路径规划为测试平台，但贡献并非新的路径求解器、特定的ALNS配置或一组特定的路径规则。贡献在于RACL方法：一种让推理智能体发现、验证、整合和解释元启发式算法控制规则的方式。在当前实验设置中，RACL在21个可行案例中的21个中改进或持平运行内存策略，在21个可行案例中的18个中改进或持平非推理的停滞触发策略，平均RACL与STP的成本差异为-0.641%。在Sevilla-9/10运行时样本中，RACL相比于固定策略平均成本降低-8.337%，相比于STP降低-1.605%，且没有显著的计算开销。在概念验证期间，Codex被用作循环中的推理智能体，观察执行过程、解释日志并提出实时有界干预。后来仅使用策略代理来使定量评估可重复。

## Abstract
This paper introduces RACL, a Reasoning-Agent Control Layer for metaheuristics. RACL places a reasoning agent above an existing optimizer. The agent does not replace the optimizer and does not modify business constraints. Instead, it controls the optimizer's internal search behavior by observing operational memory, reasoning over past behavior, formulating bounded hypotheses, testing interventions, evaluating outcomes, applying guardrails, consolidating useful policies and explaining its decisions. The experiment uses vehicle routing as a testbed, but the contribution is not a new routing solver, a particular ALNS configuration or a specific set of routing rules. The contribution is the RACL method: a way for a reasoning agent to discover, validate, consolidate and explain algorithmic control rules for a metaheuristic. In the current experimental setting, RACL improves or ties the Operational Memory Policy in 21 of 21 feasible cases and improves or ties a non-reasoning Stagnation-Triggered Policy in 18 of 21 feasible cases, with an average RACL vs STP cost delta of -0.641%. In the Sevilla-9/10 runtime sample, RACL improves average cost by -8.337% versus Fixed and -1.605% versus STP without showing material computational overhead. During the proof-of-concept, Codex was used as an in-the-loop reasoning agent observing executions, interpreting logs and proposing live bounded interventions. The policy proxy was later used only to make quantitative evaluation reproducible.

## 2. 方法论：核心思想、关键技术细节与算法流程
- **核心思想**：将推理智能体作为控制学习层，位于元启发式优化器之上。智能体不直接构造解，而是通过**有界算法动作**控制优化器的搜索行为（如调高或调低破坏强度、切换搜索模式）。业务约束（需求、车队、时间窗等）完全由优化器负责，智能体不可触碰。
- **关键技术细节**：
  - **观察状态**：每个执行块后智能体接收紧凑状态摘要：最优成本、当前成本、近期改进幅度、接受率、改进率、停滞迭代数、路线数、未分配路线、可行性指标。
  - **运行记忆存储**：历史状态、动作、理由、结果、可行性，作为证据而非真值。

  - **护栏机制**：当干预可能引入风险时（如搜索退化过大、可行性受损），立即添加限制规则。
  - **决策可解释性**：将技术动作翻译为业务语言（如“优化器卡住了，允许更强的路线重组；找到更好区域后切换到聚焦搜索”）。
- **算法流程（RACL 循环）**：
  观察（observe）→ 检索记忆（retrieve）→ 推理（reason）→ 提出有界假设（hypothesize）→ 干预（intervene）→ 评估（evaluate）→ 应用护栏（guard）→ 整合策略（consolidate）→ 解释（explain）→ 更新记忆（update memory）。
  该循环在每个执行块后重复，支持持续学习——规则可以强化、收紧、淘汰或新增。

- **方法价值**：规则本身不重要（如 route_probe 等），重要的是如何通过观察→推理→测试→护栏→整合→解释的循环自动发现和优化规则。

## 7. 优点
- **方法创新**：首次将推理智能体作为元启发式控制学习层，而不是直接生成启发式代码或端到端输出解，保留了原有优化器的职责和领域约束。
- **可解释性强**：每个决策可翻译为业务语言，适合非技术用户理解，有助于信任和合规。
- **低干预风险**：动作空间有界，且引入护栏机制防止搜索退化或不可行，保证业务约束不被破坏。
- **无需领域专家**：企业无需优化专家即可让优化器持续改进，降低了应用门槛。
- **计算开销低**：推理仅需轻量级 LLM 调用或预编译策略代理，实验中未出现显著开销。

## 8. 不足与局限
- **实验规模小**：仅两个数据集、最多 21 个案例，统计效力有限，无法证明在广泛 VRP 基准（如 CVRPLIB）上的有效性。
- **评估基线不够强**：未与最先进的神经网络控制（如图 RL 算子选择、深度强化学习 ALNS）或超启发式进化方法对比。
- **动作空间人为设计**：当前动作（route_probe 等）是作者根据经验定义的，未讨论其最优性或如何自动生成动作空间。
- **消融缺失**：没有单独分析记忆、护栏、推理步骤各自贡献多少；也没有比较不同推理模型（如 GPT-4、Claude）的效果。
- **潜在偏差**：使用 Codex 作为推理智能体可能受限于模型版本和提示工程，策略代理的泛化性未验证。
- **生产部署未实现**：概念验证后使用策略代理进行可重复评估，但真正在线运行时的延迟、LLM 成本、模型漂移等问题未讨论。
- **可推广性存疑**：仅测试了 VRP，未在排产、调度、图优化等其他元启发式应用场景验证。
